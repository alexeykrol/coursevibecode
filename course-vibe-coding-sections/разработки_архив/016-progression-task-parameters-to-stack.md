# Прогрессия: от типа задачи к стеку инструментов

## Зачем этот документ

Показать, как с ростом сложности задачи меняется рабочий стек:  
`инструмент -> обвес -> протоколы -> инфраструктура`.

## Базовые параметры любой задачи

1. **Тип запроса**: разовый Q&A или длительная работа.
2. **Контекстная зависимость**: насколько критично опираться на ваши данные, а не только на предобучение модели.
3. **Количество и разнообразие артефактов**: файлы, форматы, взаимосвязи.
4. **Объем данных**: токены, файлы, давление на контекстное окно.
5. **Временной горизонт**: одна сессия или недели/месяцы итераций.
6. **Неопределенность и ветвление**: сколько альтернатив, версий, откатов.
7. **Координация**: один человек, человек+агент, команда людей+агентов.
8. **Целевой уровень автономности**: советник, исполнитель, оркестратор.
9. **Требования к контролю/безопасности**: приватность, права доступа, аудит, комплаенс.

## Уровни стека (по возрастанию сложности)

### L0. Разовый вопрос-ответ
- **Задача**: короткий запрос, низкая цена ошибки.
- **Достаточно**: обычный чат.

### L1. Повторяемый Q&A
- **Задача**: частые однотипные запросы.
- **Добавляется**: кастомный GPT / шаблонные промпты.

### L2. Контекстная задача с небольшим набором файлов
- **Задача**: нужна опора на ваши документы, но без формального project management.
- **Добавляется**: chat-project/workspace с файлами.
- **Ограничение**: быстро упирается в объем и деградацию длинного диалога.

### L3. Профессиональная рабочая среда (IDE + файловая структура)
- **Задача**: много артефактов и связей, нужен управляемый workspace.
- **Добавляется**: IDE + файловая структура + внешняя память проекта (фреймворк).
- **Эффект**: проект = система файлов + правила, а не длинный чат.

### L4. Контроль версий, облачная синхронизация и командная работа (Git + GitHub)
- **Задача**: ветки, откаты, параллельные версии, совместная работа.
- **Добавляется**: Git + GitHub.
- **Эффект**: история изменений, diff и безопасные эксперименты.

### L5. Исполнительный режим (рост автономии агента в IDE)
- **Задача**: нужен исполнитель и массовые изменения, а не только советы.
- **Добавляется**: AI‑агент в IDE + ТЗ/критерии приемки + diff‑контроль.
- **Эффект**: переход от диалога к автономному выполнению.

### L6. Переходный уровень: простая разработка на Bolt, Lovable, v0
- **Задача**: нужен быстрый прототип/демо и интерфейсный сценарий.
- **Добавляется**: no/low‑code прототипирование.
- **Эффект**: time‑to‑demo и проверка идеи без глубокой инженерии.

### L7. Кодовые агенты в IDE: простые приложения
- **Задача**: растет сложность логики, нужны интеграции и поддерживаемость.
- **Добавляется**: полноценная кодовая база + IDE + агент + базовый деплой.
- **Эффект**: устойчивые приложения вместо хрупких прототипов.

### L8. Финальный уровень: агентные системы и следующий курс
- **Задача**: автономные процессы, память, оркестрация.
- **Добавляется**: агентные системы (RAG/knowledge, многоагентность, eval‑контуры).
- **Эффект**: переход к следующему курсу и инфраструктурному уровню.

### L9. Саморазвивающаяся агентная система (целевой горизонт)
- **Задача**: агенты создают/улучшают агентов и процессы.
- **Добавляется**: мета-оркестрация, eval-контуры, автоулучшение под политиками безопасности.

## Триггеры перехода между уровнями

Переход на следующий уровень нужен, когда выполняется хотя бы один триггер:

1. **Контекст перестал влезать** в надежный рабочий цикл.
2. **Растет число артефактов** и теряются связи между ними.
3. **Появились ветки решений** и нужна безопасная обратимость.
4. **Нужна делегация исполнения**, а не только консультации.
5. **Работа стала командной**, нужны роли и единые правила.
6. **Появились требования безопасности/комплаенса**, не закрываемые публичным SaaS.

## Короткая формула выбора стека

- Берем **минимальный достаточный уровень** под текущие параметры задачи.
- Поднимаем уровень только при явных триггерах.
- Рост сложности идет слоями:  
`инструмент -> обвес (память/ритуалы) -> протоколы -> инфраструктура`.
