# System Patterns: Архитектура курса

## Общая структура курса

### Модель построения программы
Курс следует принципу **от результата к инструментам**:
1. Показываем «до/после» и выигрыш
2. Объясняем, почему текущий подход — потолок
3. Даем инструменты как средство достижения результата
4. Закрепляем практикой с быстрыми победами

### Каркас модулей

```
Блок 1: ЗАЧЕМ? (Обоснование ценности)
   └─ Две революции: производительность + возможности
   └─ Почему чат — это тупик
   └─ Сравнение подходов (таблицы кейсов)

Блок 2: ЧТО? (Инструменты без технарщины)
   └─ IDE как «умный рабочий стол»
   └─ Git как «машина времени»
   └─ GitHub как «общий сейф»
   └─ AI-агент как «исполнитель»

Блок 3: КАК? (Практическое применение)
   └─ Быстрые победы (ускорение задач)
   └─ Единый проект (структура + версии)
   └─ Качество решений (анализ, сравнение)
   └─ Новые возможности (автоматизации, приложения)

Блок 4: ВСТРАИВАНИЕ (Персонализация)
   └─ Сценарии по ролям (маркетинг, методология, наука)
   └─ Встраивание в рабочий процесс
   └─ Дальнейшее развитие (путь к мастерству)
```

## Ключевые решения

### Методология работы
- **Роли:** SME (Subject Matter Expert — вы) + ID (Instructional Designer — агент)
- **Процесс:** Фокусированные вопросы → ответы → структурирование → итерации
- **Формат:** Модули → Уроки → Практики (1-2 на урок)

### Принципы подачи материала
1. **Минимум терминов:** «Рабочий стол» вместо «IDE», «Машина времени» вместо «система контроля версий»
2. **Максимум кейсов:** Каждая концепция иллюстрируется реальной задачей
3. **Прогрессия сложности:** От простого к сложному, без резких скачков
4. **Быстрые победы:** Результат с первого занятия

### Критические паттерны

**Паттерн 1: Триггер → Выигрыш → Инструмент**
- Сначала показываем боль и выигрыш от решения
- Только потом даем инструмент как средство

**Паттерн 2: Сравнение «Было/Стало»**
- Используем таблицы кейсов (Кейсы.html, Обоснования.html)
- Показываем конкретную ситуацию до и после

**Паттерн 3: Контекст перед действием**
- Объясняем «зачем» перед «как»
- Даем студенту осознанный выбор учиться дальше

## Компонентные отношения

### Связь файлов проекта
```
projectbrief.md (фундамент)
    ├─> productContext.md (аудитория + боли)
    ├─> systemPatterns.md (архитектура)
    └─> techContext.md (технологии)
        └─> activeContext.md (текущий фокус)
            └─> progress.md (что сделано/осталось)
```

### Исходные материалы
- `Курс по вайб кодингу...` — главный источник смыслов
- `Клиенты и проблемы.md` — портрет аудитории
- `Концепция программы курса.md` — принципы построения
- `Программа курса.md` — текущая детализация
- `Кейсы.html` / `Обоснования.html` — аргументация через сравнение

## Критические пути реализации

### Путь студента через курс
1. **Осознание проблемы** → Понимание, что чат — это потолок
2. **Принятие решения** → Готовность тратить время на изучение
3. **Первая победа** → Агент сделал задачу быстрее, чем вручную
4. **Системное понимание** → Вижу проект как целое, а не набор файлов
5. **Автономия** → Могу создавать инструменты сам

### Путь разработки курса (наш проект)
1. **Анализ материалов** → Понимание смыслов ✅
2. **Создание структуры** → Memory Bank + модули (в процессе)
3. **Детализация уроков** → Разбивка модулей на уроки
4. **Наполнение контентом** → Тексты, примеры, практики
5. **Тестирование** → Пилот с реальными студентами
