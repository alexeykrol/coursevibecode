# Claude Code: как устроены `CLAUDE.md`, `rules` и `settings.json`

Этот документ объясняет, как в Claude Code устроены уровни инструкций и настроек:

- что задается через `CLAUDE.md`;
- что задается через `settings.json`;
- как работают проектные и личные уровни;
- как настраивать отдельные подпапки;
- как не запутаться в приоритетах.

Документ написан простым языком для людей без инженерного бэкграунда.

Проверено по официальной документации Claude Code на 20 марта 2026 года.

Важно: в разговоре люди иногда говорят "Cloud Markdown" или "Claude Markdown". В официальной документации правильное имя файла: `CLAUDE.md`.

## Короткий ответ за 1 минуту

В Claude Code есть два разных механизма:

1. `CLAUDE.md` и `.claude/rules/`
Это слой инструкций и памяти. Он говорит Claude, как лучше работать, что учитывать, какие правила соблюдать, какой стиль использовать.

2. `settings.json`
Это слой реальных настроек клиента. Он управляет правами, sandbox, plugins, hooks, MCP, языком ответов и другими параметрами среды.

Если совсем коротко:

- хотите управлять моим поведением, стилем и контекстом: используйте `CLAUDE.md` и `rules`;
- хотите жестко управлять правами, разрешениями и техническими режимами: используйте `settings.json`.

## Самая важная мысль

`CLAUDE.md` не является "жесткой конфигурацией". Это инструкции, которые попадают в контекст.

`settings.json` уже является настоящей конфигурацией. Если в настройках что-то запрещено, Claude не сможет это обойти просто потому, что в `CLAUDE.md` написано иначе.

Поэтому хороший практический вопрос всегда звучит так:

"Я хочу дать агенту совет?"  
Тогда `CLAUDE.md`.

"Я хочу реально разрешить, запретить или ограничить?"  
Тогда `settings.json`.

## Общая карта файлов

Ниже базовая карта уровней, которую удобно держать в голове.

```text
Managed / organization level
├── Managed CLAUDE.md
└── managed settings

User level
└── ~/.claude/
    ├── CLAUDE.md
    ├── settings.json
    ├── rules/
    ├── skills/
    └── agents/

Project level
project-root/
├── CLAUDE.md
├── .mcp.json
└── .claude/
    ├── CLAUDE.md
    ├── settings.json
    ├── settings.local.json
    ├── rules/
    ├── skills/
    └── agents/

Subdirectory level
project-root/subdir/
└── CLAUDE.md
```

## 1. Как устроен `CLAUDE.md`

### Что это такое

`CLAUDE.md` дает Claude постоянные инструкции:

- как устроен проект;
- какие команды запускать;
- какие правила важны;
- какой стиль и workflow приняты;
- что в этом проекте считается нормой.

Это не "магическая настройка", а понятный текст с правилами.

### Где он может лежать

Официально документация выделяет такие уровни:

1. Managed `CLAUDE.md`
Для всей организации или для всех машин компании.

2. `~/.claude/CLAUDE.md`
Ваши личные глобальные инструкции для всех проектов.

3. `./CLAUDE.md` или `./.claude/CLAUDE.md`
Общие инструкции проекта.

4. `subdir/CLAUDE.md`
Более локальные инструкции для конкретной части проекта.

### Как он загружается

Claude Code идет вверх по дереву папок от текущей рабочей директории и загружает `CLAUDE.md` по пути.

Пример:

```text
repo/
├── CLAUDE.md
└── packages/
    ├── CLAUDE.md
    └── frontend/
        └── CLAUDE.md
```

Если вы запускаете Claude Code в `repo/packages/frontend`, он может учитывать:

- `repo/CLAUDE.md`
- `repo/packages/CLAUDE.md`
- `repo/packages/frontend/CLAUDE.md`

Чем ближе файл к текущей папке, тем он более специфичен.

### Что происходит с подпапками

Есть важный нюанс:

- `CLAUDE.md` в папках выше текущей рабочей директории загружаются сразу при старте;
- `CLAUDE.md` в подпапках ниже текущей директории подхватываются по мере работы, когда Claude читает файлы в этих подпапках.

Это удобно для больших репозиториев: не нужно грузить весь зоопарк инструкций сразу.

### Что лучше хранить в `CLAUDE.md`

Хорошо подходят:

- обзор проекта;
- архитектурные договоренности;
- build/test команды;
- naming conventions;
- workflow команды;
- требования к ответам и оформлению;
- правила ревью;
- то, что должно понимать любой человек или агент в проекте.

Плохо подходят:

- чувствительные секреты;
- жесткие запреты доступа;
- технические ограничения, которые должны enforce'иться клиентом;
- слишком длинные полотна текста.

Официальная рекомендация Claude Code: держать один `CLAUDE.md` примерно до 200 строк, а большие инструкции дробить через imports или `.claude/rules/`.

### `CLAUDE.md` в корне или `.claude/CLAUDE.md`?

Официально поддерживаются оба варианта.

Практически лучше выбрать один стиль на проект:

- либо `project-root/CLAUDE.md`;
- либо `project-root/.claude/CLAUDE.md`.

Не стоит без причины держать оба сразу. Это почти всегда увеличивает путаницу.

## 2. Как устроены `.claude/rules/`

### Зачем они нужны

Если один большой `CLAUDE.md` начинает разрастаться, Claude Code рекомендует выносить правила в `.claude/rules/`.

Это удобно, когда:

- в проекте много разных тем;
- есть frontend и backend с разными правилами;
- есть markdown-файлы, API, тесты, docs и для каждого слоя нужен свой набор требований.

### Два типа rules

1. Общие rules
Если в rule-файле нет `paths`, он грузится всегда.

2. Path-scoped rules
Если в YAML frontmatter задано `paths`, правило срабатывает только когда Claude работает с файлами, подходящими под этот паттерн.

Пример:

```md
---
paths:
  - "src/api/**/*.ts"
---

# API Rules

- Все эндпоинты должны проверять входные данные.
- Ошибки должны возвращаться в стандартном формате.
```

### Где rules могут жить

- `~/.claude/rules/` — ваши личные глобальные правила;
- `project/.claude/rules/` — правила проекта;
- по документации project rules имеют больший приоритет, чем user rules.

### Когда rules лучше, чем nested `CLAUDE.md`

Если задача звучит как:

"Мне нужны разные правила для разных типов файлов"

то лучше использовать `rules`.

Если задача звучит как:

"У этой подпапки отдельная логика, отдельный процесс, отдельный контекст"

то уместен nested `CLAUDE.md`.

## 3. Как устроен `settings.json`

### Что это такое

`settings.json` управляет реальной конфигурацией Claude Code.

Через него задают:

- permissions;
- sandbox;
- hooks;
- plugins;
- MCP;
- env;
- язык ответов;
- модели;
- fast mode и другие настройки клиента.

### Основные уровни настроек

Официальные scopes такие:

1. Managed
Централизованные настройки организации. Самый высокий уровень.

2. User
`~/.claude/settings.json`

3. Project
`project/.claude/settings.json`

4. Local
`project/.claude/settings.local.json`

### Приоритет уровней

Когда один и тот же параметр задан в нескольких местах, работает такой порядок:

1. Managed
2. CLI arguments
3. Local
4. Project
5. User

Это значит:

- local настройки проекта сильнее project;
- project сильнее user;
- managed сильнее всего;
- временные CLI overrides стоят выше project и user, но ниже managed.

### Что обычно класть в каждый уровень

#### `~/.claude/settings.json`

Сюда хорошо класть:

- ваши глобальные предпочтения;
- язык ответов;
- модели по умолчанию;
- user-level plugins;
- user-level permissions, которые удобны почти везде.

#### `project/.claude/settings.json`

Сюда хорошо класть:

- общие project permissions;
- hooks команды;
- team plugins;
- project-level MCP policy;
- team-shared language or output settings;
- другие настройки, которые должны быть одинаковыми у всей команды.

#### `project/.claude/settings.local.json`

Это ваш личный слой только для этого проекта.

Сюда хорошо класть:

- машинно-специфичные override;
- временные эксперименты;
- персональные исключения;
- `claudeMdExcludes`;
- личные local permissions для этого проекта.

Документация отдельно отмечает, что Claude Code автоматически добавляет `.claude/settings.local.json` в git ignore при создании.

### Как ведут себя массивы

В `settings.json` массивы часто не просто заменяются, а merge'ятся и dedupe'ятся.

Это особенно важно для:

- allowlists;
- hook URL allowlists;
- `claudeMdExcludes`;
- некоторых MCP и plugin списков.

Поэтому более узкий уровень может не только заменить настройку, но и добавить свои элементы к существующему набору.

### Как работают permissions

Permission rules проверяются в таком порядке:

1. `deny`
2. `ask`
3. `allow`

Первое совпавшее правило выигрывает.

Значит, если что-то разрешено на user level, но запрещено на project level, запрет победит.

Это важнейшее правило безопасности.

### Реальный инцидент: как `settings.local.json` превратился в хранилище кредов

Этот риск не теоретический.

В реальной сессии проблема была замечена случайно: пользователь просто открыл `.claude/settings.local.json` в IDE и увидел внутри не только настройки, но и чувствительные данные.

В файле оказались:

- полный приватный SSH-ключ;
- service role токен;
- IP-адрес, порт и имя пользователя сервера;
- десятки разрешенных командных строк, в которые эти значения были встроены прямо inline.

Почему так произошло:

- Claude Code умеет запоминать разрешения, чтобы не спрашивать вас каждый раз заново;
- когда пользователь нажимает `Allow`, в локальные разрешения может сохраняться сама строка команды;
- если в строке команды уже были ключ, токен, IP, порт или имя пользователя, то они и попадают в `.claude/settings.local.json`.

Это важно понять правильно:

- это не классический баг в духе "система сделала что-то совсем не то";
- это ожидаемое поведение механизма разрешений, если пользователь одобряет команды с чувствительными данными прямо в строке.

### Практический вывод из инцидента

Безопасная схема должна выглядеть так:

1. Разрешения отдельно, креды отдельно.
В `.claude/settings.local.json` должны жить только паттерны разрешений, а не реальные ключи и токены.

2. Никаких `Allow` на команды с кредами inline.
Если в строке уже есть токен, пароль, приватный ключ, IP, логин или connection string, сначала вынесите секреты в отдельный локальный файл, например `.production-credentials`, и только потом решайте вопрос с разрешением.

3. `.claude/settings.local.json` должен быть защищен на уровне репозитория.
Не только через локальный `.git/info/exclude`, а через `.gitignore`, чтобы это правило жило в кодовой базе.

4. Локальные разрешения надо иногда вручную проверять.
Если файл ни разу не открывать, в нем может долго накапливаться опасный мусор.

### Какой паттерн считать правильным

Правильно:

```text
"Bash(ssh:*)"
"Bash(mariadb:*)"
```

Неправильно:

```text
"Bash(ssh -i ~/.ssh/prod_key -p 65002 user@1.2.3.4 ...)"
"Bash(curl -H 'Authorization: Bearer <token>' ...)"
```

В первом случае вы разрешаете тип действия.
Во втором случае вы превращаете файл разрешений в склад чувствительных данных.

## 4. Что важно знать про `CLAUDE.local.md`

Это как раз та зона, где у людей часто возникает путаница.

На момент проверки документации `CLAUDE.local.md` не описан как официальный стандартный слой memory в Claude Code.

Официально documentation говорит про:

- managed `CLAUDE.md`;
- `~/.claude/CLAUDE.md`;
- `./CLAUDE.md`;
- `./.claude/CLAUDE.md`;
- nested `CLAUDE.md` в поддиректориях;
- `.claude/rules/`.

Если вам нужны личные инструкции для одного проекта, но вы не хотите коммитить их в git, более официальный путь выглядит так:

1. Держать shared project `CLAUDE.md`.
2. При необходимости импортировать в него ваш локальный файл из домашней папки через `@~/.claude/...`.

Это прямо описано в документации как способ подключать личные project-specific preferences.

## 5. Что важно знать про `settings.local.json`

Вот тут наоборот есть официальный локальный слой, но только на уровне проекта:

- `project/.claude/settings.local.json`

Именно он предназначен для:

- личных overrides внутри конкретного репозитория;
- машинно-специфичных настроек;
- временных экспериментов.

Отдельный `~/.claude/settings.local.json` как стандартный официальный scope в docs не описан.

## 6. Связанные файлы, про которые полезно знать

### `.mcp.json`

Project-scoped MCP серверы хранятся в `.mcp.json` в корне проекта.

Это важно, потому что MCP не живет только в `settings.json`.

### `~/.claude.json`

Часть состояния Claude Code хранится не в `settings.json`, а в `~/.claude.json`.

Там живут, среди прочего:

- user and local MCP configuration;
- session state;
- trust state;
- интерфейсные preferences;
- различные кэши.

### `~/.claude/skills/` и `project/.claude/skills/`

Skills — это не то же самое, что `CLAUDE.md`.

Их удобно использовать для повторяемых задач:

- код-ревью;
- разбор архитектуры;
- генерация отчета;
- подготовка release notes;
- специальные команды для конкретного проекта.

Документация отдельно описывает enterprise, personal и project skills, а также автоматическое обнаружение nested skills из подпапок, например:

```text
packages/frontend/.claude/skills/
```

### Auto memory

Auto memory — это отдельный механизм, не то же самое, что `CLAUDE.md`.

`CLAUDE.md` пишете вы.  
Auto memory Claude пишет сам.

Auto memory хранится в:

```text
~/.claude/projects/<project>/memory/
```

Важно:

- у каждого git-репозитория своя memory directory;
- все worktree и подпапки внутри одного repo используют одну и ту же память;
- это локальная память машины, а не shared team memory.

### `claudeMdExcludes`

Если вы работаете в большом monorepo, вам могут случайно мешать чужие родительские `CLAUDE.md`.

Для этого есть `claudeMdExcludes`.

Официальная рекомендация docs: хранить такие исключения в `.claude/settings.local.json`, чтобы исключение было личным, а не командным.

## 7. Как я бы рекомендовал строить структуру в обычном проекте

### Вариант A. Небольшой одиночный проект

```text
project/
├── CLAUDE.md
├── .mcp.json
└── .claude/
    ├── settings.json
    ├── settings.local.json
    ├── rules/
    └── skills/
```

Логика:

- `CLAUDE.md` — общее описание и workflow;
- `.claude/settings.json` — общие project settings;
- `.claude/settings.local.json` — ваши локальные overrides;
- `.claude/rules/` — модульные правила;
- `.claude/skills/` — повторяемые команды.

### Вариант B. Большой проект или monorepo

```text
repo/
├── CLAUDE.md
├── .mcp.json
├── .claude/
│   ├── settings.json
│   ├── settings.local.json
│   ├── rules/
│   └── skills/
├── packages/
│   ├── CLAUDE.md
│   ├── frontend/
│   │   ├── CLAUDE.md
│   │   └── .claude/
│   │       └── skills/
│   └── backend/
│       └── CLAUDE.md
```

Логика:

- root `CLAUDE.md` — общие правила всей репы;
- `packages/CLAUDE.md` — правила уровня пакетов;
- `packages/frontend/CLAUDE.md` — более узкий контекст для frontend;
- nested skills — если отдельный пакет требует своих инструментов;
- `.claude/rules/` — если нужно фильтровать по типу файлов, а не только по папке.

## 8. Playbook: кейсы простым языком

Ниже не абстракция, а типовые ситуации в формате "я хочу получить вот это".

### Кейс 1. Я хочу, чтобы Claude во всех проектах отвечал мне по-русски и в моем любимом стиле

Проблема:
вы не хотите на каждом проекте заново объяснять язык, формат ответа и базовые личные предпочтения.

Что сделать:

1. Положить ваши общие инструкции в `~/.claude/CLAUDE.md`.
2. Положить глобальные технические настройки в `~/.claude/settings.json`.

Что именно туда класть:

- в `CLAUDE.md`: стиль объяснения, язык, любимый формат плана, предпочтения по объяснениям;
- в `settings.json`: `language`, модель по умолчанию, глобальные user plugins.

Когда это правильно:
когда правило должно работать почти везде.

### Кейс 2. Я хочу, чтобы у всей команды в одном проекте были одинаковые правила

Проблема:
иначе каждый агент и каждый участник команды будут работать по-разному.

Что сделать:

1. Создать `project/CLAUDE.md`.
2. Создать `project/.claude/settings.json`.

Что именно туда класть:

- в `CLAUDE.md`: архитектуру, naming, workflow, build/test команды;
- в `.claude/settings.json`: project permissions, hooks, team plugins, project MCP.

Когда это правильно:
когда правило нужно разделять через git со всей командой.

### Кейс 3. Я хочу в одном проекте свои личные overrides, но не хочу засорять git

Проблема:
вам нужна личная настройка только для этого репозитория.

Что сделать:

1. Использовать `project/.claude/settings.local.json`.

Что туда класть:

- personal permissions;
- временные эксперименты;
- `claudeMdExcludes`;
- machine-specific настройки.

Когда это правильно:
когда это должно работать только у вас и только в одном проекте.

### Отдельный кейс безопасности. Я хочу, чтобы разрешения не превращались в хранилище кредов

Проблема:
при нажатии `Allow` на полную строку команды Claude Code может сохранить в `.claude/settings.local.json` именно эту строку, вместе с ключами, токенами, IP и другими чувствительными аргументами.

Что сделать:

1. Откройте `.claude/settings.local.json` и проверьте, нет ли там секретов inline.
2. Если видите ключи, токены, пароли или connection strings, вынесите реальные значения в отдельный локальный файл, например `.production-credentials`.
3. Оставьте в разрешениях только общие паттерны, а не полные командные строки с секретами.
4. Убедитесь, что `.claude/settings.local.json` защищен в `.gitignore` репозитория, а не только локально на одной машине.

Когда это правильно:
всегда, как только агент начинает регулярно работать с ssh, curl, базами данных, cloud CLI или прод-серверами.

### Кейс 4. У меня monorepo, и frontend и backend требуют разных правил

Проблема:
один общий `CLAUDE.md` начинает мешать. Он или слишком общий, или слишком длинный.

Что сделать:

Вариант 1:
использовать nested `CLAUDE.md` в отдельных подпапках.

Вариант 2:
использовать `.claude/rules/` с `paths`.

Когда лучше nested `CLAUDE.md`:
если у подпапки реально свой контекст, свой workflow и свои соглашения.

Когда лучше `rules`:
если вы хотите нацеливаться не только на папку, но и на типы файлов.

### Кейс 5. Я не хочу, чтобы Claude подтягивал нерелевантные инструкции из родительской папки

Проблема:
в большом monorepo над вашей папкой может лежать чужой `CLAUDE.md`, который только мешает.

Что сделать:

1. Добавить `claudeMdExcludes` в `project/.claude/settings.local.json`.

Когда это правильно:
когда исключение должно быть личным и не должно ломать работу другим.

### Кейс 6. Я хочу личные project-specific инструкции, но не хочу коммитить их

Проблема:
общий project `CLAUDE.md` нужен команде, но есть ваши личные правила для этого конкретного проекта.

Что сделать:

1. Оставить shared project `CLAUDE.md`.
2. Импортировать в него локальный файл из домашней папки, например через `@~/.claude/my-project-notes.md`.

Когда это правильно:
когда нужна именно личная project memory, но без отдельного неофициального слоя типа `CLAUDE.local.md`.

### Кейс 7. Я хочу запускать повторяемые команды для проекта одной и той же логикой

Проблема:
каждый раз заново писать длинный запрос неудобно.

Что сделать:

1. Создать project skills в `project/.claude/skills/`.

Примеры:

- обзор проекта;
- выпускной checklist;
- подготовка changelog;
- редакторский проход;
- review сценарий.

Когда это правильно:
когда задача повторяется и ее лучше превращать в именованный инструмент.

### Кейс 8. Я хочу, чтобы конкретный пакет monorepo имел свои skills

Проблема:
у `frontend` и `backend` разные повторяемые задачи.

Что сделать:

1. Положить nested skills в `packages/frontend/.claude/skills/`.

Когда это правильно:
когда разные части проекта реально живут своей жизнью.

### Кейс 9. Я хочу shared MCP для проекта

Проблема:
команда должна видеть одинаковые project MCP servers.

Что сделать:

1. Держать project-scoped MCP в `.mcp.json`.
2. Если нужно, дополнительно ограничивать и разрешать через `settings.json`.

Когда это правильно:
когда интеграция должна быть общей для команды, а не только вашей локальной.

### Кейс 10. Я хочу добавить внешнюю общую папку с инструкциями

Проблема:
часть инструкций или общих документов лежит вне проекта.

Что сделать:

1. Подключить директорию через `--add-dir`.
2. Если хотите, чтобы из нее грузились `CLAUDE.md` и `rules`, включить `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1`.

Когда это правильно:
когда есть shared knowledge base вне репозитория.

## 9. Как проверять, что все действительно работает

### Проверить memory слой

Используйте `/memory`.

Эта команда помогает:

- увидеть, какие `CLAUDE.md` и `rules` реально загружены;
- включить или выключить auto memory;
- открыть memory files.

### Проверить settings слой

Используйте `/config`.

Это официальный вход в настройки Claude Code и проверку активной конфигурации.

### Если Claude ведет себя не так, как ожидалось

Сначала проверьте:

1. этот файл реально лежит в поддерживаемом месте;
2. он реально попадает в текущую область видимости;
3. нет ли конфликтующих инструкций в другом `CLAUDE.md`;
4. не запрещает ли нужное действие `settings.json`;
5. не загружается ли лишний родительский `CLAUDE.md`.

## 10. Рекомендуемая схема по умолчанию

Если не хочется усложнять, вот безопасный базовый набор:

```text
~/.claude/
├── CLAUDE.md
└── settings.json

project/
├── CLAUDE.md
├── .mcp.json
└── .claude/
    ├── settings.json
    ├── settings.local.json
    ├── rules/
    └── skills/
```

Этого уже хватает для большинства проектов.

Добавляйте nested `CLAUDE.md` только тогда, когда реально появляется отдельный подпроект с отдельными правилами.

## 11. Что не делать

Ниже короткий список самых частых ошибок.

### Не пытайтесь решать все одним файлом

Если `CLAUDE.md` раздулся, разносите:

- общие инструкции — в основной `CLAUDE.md`;
- точечные правила — в `.claude/rules/`;
- повторяемые сценарии — в skills;
- реальные ограничения — в `settings.json`.

### Не храните жесткие запреты только в `CLAUDE.md`

Если что-то действительно нельзя, это должно жить в `settings.json`, а не только в текстовой инструкции.

### Не стройте архитектуру на неофициальных слоях

Если слой не описан в актуальной документации, не делайте его опорой всей системы.

### Не держите два project `CLAUDE.md` без ясной причины

Выберите либо `./CLAUDE.md`, либо `./.claude/CLAUDE.md`.

### Не путайте project settings и personal local overrides

Если настройка нужна всем, это `project/.claude/settings.json`.

Если настройка нужна только вам, это `project/.claude/settings.local.json`.

### Не превращайте `.claude/settings.local.json` в склад кредов

Если вы жмете `Allow` на команды с чувствительными данными inline, файл разрешений может начать хранить то, что вообще не должно там жить.

Разрешения должны хранить паттерны действий.
Креды должны жить отдельно.

## Глоссарий

**Managed settings**  
Организационные настройки, которые централизованно задаются компанией и не могут быть переопределены обычным пользователем.

**User scope**  
Личный уровень, который действует для всех ваших проектов.

**Project scope**  
Уровень конкретного репозитория, который обычно коммитится и разделяется с командой.

**Local scope**  
Ваш личный уровень внутри одного конкретного проекта. Обычно не коммитится.

**`CLAUDE.md`**  
Файл с инструкциями и постоянным контекстом для Claude Code.

**Rules**  
Модульные markdown-правила в `.claude/rules/`, которые могут быть общими или path-scoped.

**Path-scoped rule**  
Правило, которое срабатывает только для определенных файлов или папок по glob-паттерну.

**Skill**  
Повторяемый именованный инструмент или сценарий работы для Claude Code.

**MCP**  
Механизм подключения внешних серверов и инструментов к Claude Code.

**Auto memory**  
Автоматическая локальная память проекта, которую Claude ведет сам.

**Lazy-load**  
Отложенная загрузка. В контексте `CLAUDE.md` это значит, что некоторые файлы подхватываются не при старте, а в момент, когда Claude начинает работать с нужной подпапкой.

**Import через `@path`**  
Способ подключить дополнительный файл внутрь `CLAUDE.md`.

**Monorepo**  
Большой репозиторий, в котором лежит несколько проектов или пакетов.
