# Матрица задач и кейсов курса (уровни L3-L8, где L6 переходный, L7 IDE, L8 финальный)

## Как читать этот документ

- Каждый уровень содержит:
1. **Штатные задачи уровня** — что студент должен уметь делать текущим стеком.
2. **Штатные кейсы** — узнаваемые практические задачи с ожидаемым результатом.
3. **Миссии-препятствия** — задачи, которые упираются в потолок уровня и требуют перехода выше.
- Формат препятствия:
`Задача -> Упор на текущем уровне -> Выросший параметр -> Переход`.

## Границы этой версии

- Уровень `L2` (чатовые задачи и knowledge-workspace) покрыт бесплатным курсом и здесь не повторяется.
- Ядро текущего курса: `L3`, `L4`, `L5` (без задач разработки приложений).
- Переходный уровень `L6` включен как мост от no‑code к IDE.
- Уровень `L7` — простые приложения в IDE с кодовыми агентами (Claude Code/Codex).
- Финальный уровень `L8` — обзор будущего и логика перехода в платный курс по агентам.
- Начальные уровни (`L0-L1`) и глубокие инженерные уровни (`L9+`) исключены.

## L3. Профессиональная рабочая среда (IDE + файловая структура)

### Итог уровня (для преподавателя)

Инструменты: IDE (VS Code), файловая структура проекта, контекстная память/фреймворк (например, `/Users/alexeykrolmini/Downloads/Code/Project_init`).  
Что меняется/результат: студент превращает хаос документов в управляемую структуру и закрепляет память проекта между сессиями.  
Примеры задач (из кейсов уровня): разложить курс на модули, собрать юридическое досье, создать шаблоны рабочих документов.

### Что показывать на уровне (преподавателю)

Что показать: как устроен workspace, навигация и быстрый поиск по 50+ файлам, базовая структура папок проекта.  
Инструменты: VS Code (или аналог), файловая система, memory‑фреймворк (`/Users/alexeykrolmini/Downloads/Code/Project_init`).  
Навык: структурировать проект и фиксировать контекст/правила в явном виде.  
Концепция: проект = система файлов + внешняя память, а не длинный чат.

### Штатные задачи уровня (миссии развития)

1. Разложить большой курс или книгу на управляемую структуру папок и файлов.
2. Настроить быстрый поиск и навигацию по 50+ документам.
3. Собрать единую библиотеку источников: статьи, таблицы, иллюстрации, ссылки.
4. Систематизировать интервью, расшифровки и полевые заметки исследования.
5. Подготовить файловую архитектуру для юридического или медицинского досье.
6. Свести материалы маркетинговой кампании в единый workspace.
7. Вести несколько версий презентации и методички без дублирования хаоса.
8. Создать набор шаблонов документов для повторяемых задач команды.
9. Унифицировать терминологию и форматирование во всех ключевых текстах.
10. Настроить понятные правила именования и хранения файлов для команды.

### Штатные кейсы (задача / ожидаемые результаты)

1. **Задача:** Разложить курс на модули, уроки и материалы в единой структуре папок. **Ожидаемые результаты:** прозрачная карта проекта и быстрый доступ к артефактам.
2. **Задача:** Собрать книгу как проект (главы, приложения, ссылки на источники). **Ожидаемые результаты:** управляемая структура вместо разрозненных документов.
3. **Задача:** Организовать исследовательские интервью и транскрипты по темам. **Ожидаемые результаты:** каталог интервью и быстрый поиск цитат по смысловым блокам.
4. **Задача:** Подготовить юридическое досье клиента из десятков файлов. **Ожидаемые результаты:** структурный набор документов по этапам и типам рисков.
5. **Задача:** Свести материалы маркетинговой кампании (посты, письма, лендинг-тексты). **Ожидаемые результаты:** единый контент-пакет без потерь версий.
6. **Задача:** Найти и унифицировать терминологию в 70 документах. **Ожидаемые результаты:** единый словарь и снижение смысловых противоречий.
7. **Задача:** Нормализовать форматирование базы знаний команды. **Ожидаемые результаты:** читаемая, единообразная документация.
8. **Задача:** Создать шаблоны для брифов, отчетов и протоколов встреч. **Ожидаемые результаты:** повторяемый формат рабочих документов.
9. **Задача:** Настроить правила именования файлов для многоавторского проекта. **Ожидаемые результаты:** меньше дублей и потерь документов.
10. **Задача:** Подготовить workspace для учебной программы на 6 месяцев. **Ожидаемые результаты:** понятная навигация для авторов, методистов и редакторов.

### Стек уровня

- VS Code (или другая IDE).
- Локальная файловая система проекта.
- Базовые расширения редактора.
- Контекстная память и правила проекта (например, `/Users/alexeykrolmini/Downloads/Code/Project_init`).

### Подходы уровня

1. «Один проект — один workspace — одна структура папок».
2. Сначала структура и навигация, потом массовые изменения.
3. Фиксировать контекст и правила проекта через внешний memory/фреймворк.
4. Не хранить проект как хаотичный набор документов.

### Параметры эскалации (триггеры перехода)

- Нужны безопасные откаты и ветки.
- Нужен аудит изменений и релизные версии.
- Несколько человек правят один проект и нужна синхронизация.

### Миссии-препятствия (требуют следующий уровень)

1. **Массово переименовать термин в 120 файлах.** -> Ошибка правки без безопасного отката. -> Риск изменений. -> **L4**.
2. **Перестроить структуру курса по папкам.** -> Нельзя легко восстановить вчерашнее состояние. -> История состояния. -> **L4**.
3. **Сделать две версии продающей страницы (контент).** -> Нет нормального механизма параллельных веток. -> Ветвление. -> **L4**.
4. **Жить на Ctrl+Z и автосейвах.** -> Память только «на сейчас», без долговременной истории и веток. -> Временной горизонт. -> **L4**.
5. **Экспериментировать с новым стилем книги.** -> Страшно ломать основной вариант. -> Безопасность экспериментов. -> **L4**.
6. **Работать с помощником над теми же файлами.** -> Конфликты и перезаписи без прозрачного merge. -> Координация. -> **L4/L7**.
7. **Вести юридические черновики.** -> Нужен аудит изменений для доверия и проверки. -> Трассируемость. -> **L4**.
8. **Выпустить версию методички клиенту.** -> Нет четкой фиксации релизной версии. -> Релизный контроль. -> **L4**.
9. **Поддерживать несколько клиентских вариантов.** -> Копии `final_final` множат хаос. -> Вариативность. -> **L4**.
10. **Вернуть состояние после неудачных правок.** -> Точечный откат почти невозможен без потерь. -> Обратимость. -> **L4**.

## L4. Контроль версий, облачная синхронизация и командная работа (Git + GitHub)

### Итог уровня (для преподавателя)

Инструменты: Git + GitHub (ветки, diff, PR, review).  
Что меняется/результат: студент получает историю изменений, откаты и совместную работу без потерь.  
Примеры задач (из кейсов уровня): сравнить версии договора, вести альтернативную ветку рекламных сообщений, восстановить проект через clone.

### Что показывать на уровне (преподавателю)

Что показать: commit, diff, ветка, откат, push/pull, PR и review на реальном материале.  
Инструменты: Git локально, GitHub как синхронизация и командная работа.  
Навык: контроль версий, безопасные эксперименты, совместная работа без перезаписей.  
Концепция: история изменений и ветвление — основа управляемости, а не «финал‑финал‑v7».

### Штатные задачи уровня (миссии развития)

1. Фиксировать изменения методичек, курсов и политик понятными commit-ами.
2. Сравнивать версии документов перед публикацией или отправкой клиенту.
3. Вести параллельные ветки для альтернативных редакций и стратегий.
4. Синхронизировать проект между несколькими устройствами и участниками команды.
5. Восстанавливать рабочую копию проекта из GitHub в случае потерь.
6. Организовать совместную правку текстов без перезаписи чужих изменений.
7. Отмечать релизные версии пакетов материалов тегами.
8. Откатывать неудачные правки без разрушения всей рабочей ветки.
9. Хранить историю, кто и когда менял критичные документы.
10. Поддерживать прозрачный review-процесс для контента и аналитики.

### Штатные кейсы (задача / ожидаемые результаты)

1. **Задача:** Фиксировать ежедневный прогресс курса commit-ами. **Ожидаемые результаты:** прозрачная история решений и быстрый возврат к рабочей версии.
2. **Задача:** Сравнить две версии договора перед отправкой юристу. **Ожидаемые результаты:** четкий diff и контроль критичных правок.
3. **Задача:** Подготовить альтернативную редакцию рекламных сообщений в отдельной ветке. **Ожидаемые результаты:** безопасное тестирование без риска для основного варианта.
4. **Задача:** Восстановить проект на новом ноутбуке через clone. **Ожидаемые результаты:** полная рабочая копия без ручной пересылки файлов.
5. **Задача:** Выпустить стабильную версию материалов для клиента по тегу релиза. **Ожидаемые результаты:** зафиксированный пакет и понятный changelog.
6. **Задача:** Организовать совместную правку с редактором и аналитиком. **Ожидаемые результаты:** контролируемое слияние без потери правок.
7. **Задача:** Откатить ошибочную массовую замену терминов. **Ожидаемые результаты:** быстрое восстановление корректного состояния.
8. **Задача:** Вести историю изменений политики качества сервиса. **Ожидаемые результаты:** трассируемость ответственности по изменениям.
9. **Задача:** Проверять входящие PR с правками учебного контента. **Ожидаемые результаты:** стабильное качество перед публикацией.
10. **Задача:** Поддерживать архив версий исследовательского отчета. **Ожидаемые результаты:** понятная эволюция выводов и источников.

### Стек уровня

- Git локально.
- GitHub как удаленный сейф/центр синхронизации и командной работы.
- Базовый workflow commit/push/pull/branch.

### Подходы уровня

1. Commit маленькими логическими порциями.
2. Любой рискованный эксперимент — в отдельной ветке.
3. Регулярная синхронизация и прозрачный diff перед merge.
4. Командная работа через PR и review как базовая дисциплина.

### Параметры эскалации (триггеры перехода)

- Ручные изменения стали узким местом скорости.
- Требуется исполнитель, а не только контроль версий.
- Нужны системные массовые правки и автономные циклы.

### Миссии-препятствия (требуют следующий уровень)

1. **Обновить 300 файлов под новый оффер.** -> Без агента слишком много ручной работы. -> Объем исполнения. -> **L5**.
2. **Работать только с локальным Git без GitHub.** -> Риск потери, нет надежной синхронизации между устройствами. -> Надежность/координация. -> **L4 (добавить GitHub как обязательный слой)**.
3. **Поддерживать контент на 2 языках.** -> Синхронизация версий и правок становится рутиной. -> Мультиязычный масштаб. -> **L5**.
4. **Рефакторить большой текстовый проект.** -> Человек медленно находит все зависимые места. -> Связность артефактов. -> **L5**.
5. **Делать регулярные аналитические дайджесты.** -> Ручной пайплайн съедает часы каждую неделю. -> Рутинные операции. -> **L5**.
6. **Поддерживать SOP в актуальном виде.** -> Нужен исполнитель, который проходит по связанным файлам. -> Объем и регулярность. -> **L5**.
7. **Выпускать персонализированные пакеты материалов для 8 сегментов.** -> Ручная сборка медленная и ошибкоопасная. -> Масштаб вариативности. -> **L5**.
8. **Проверять большие правки от подрядчика/чата.** -> Нужен режим «доверяй, но проверяй» через системный diff + автопроверки. -> Контроль качества. -> **L5**.
9. **Вести научную статью с правками рецензентов.** -> Слияние веток и повторные проходы занимают слишком много времени вручную. -> Цикл ревизий. -> **L5**.
10. **Собирать квартальный комплаенс-отчет из разных подразделений.** -> Ручное сведение версий и форматов ломает сроки. -> Координационная сложность. -> **L5**.

## L5. Исполнительный режим (рост автономии агента в IDE)

Агенты появляются уже на L2; на L5 фокус смещается на автономию.  
Здесь мы перестаем «вести диалог шаг за шагом» и начинаем проектировать работу так, чтобы агент делал максимум самостоятельно, а человек контролировал результат.

### Итог уровня (для преподавателя)

Инструменты: IDE + Git/GitHub + AI‑агент, правила постановки задач (ТЗ, критерии приемки, чеклисты).  
Что меняется/результат: студент переходит от микроменеджмента к автономному исполнению с контролем качества.  
Примеры задач (из кейсов уровня): массово обновить оффер, унифицировать стиль 40 уроков, классифицировать входящую почту.

### Что показывать на уровне (преподавателю)

Что показать: постановка автономного ТЗ, пакетное поручение агенту, проверка результата через diff и чеклисты.  
Инструменты: IDE, GitHub, AI‑агент (Cline/Claude Code и аналоги).  
Навык: делегирование с контролем качества вместо пошагового диалога.  
Концепция: автономия агента = правильная постановка задачи + прозрачная проверка результата.

### Штатные задачи уровня (миссии развития)

1. Формулировать автономные ТЗ и критерии приемки для агента.
2. Делать пакетные поручения вместо пошагового диалога.
3. Организовать контроль качества через diff, чеклисты и автоматические проверки.
4. Поручать агенту массовую переработку текстов в десятках файлов.
5. Автоматически приводить контент к единому тону и структуре.
6. Анализировать большие массивы клиентской обратной связи.
7. Нормализовать большие таблицы и отчеты для аналитики.
8. Готовить персонализированные материалы под разные сегменты аудитории.
9. Собирать пакет документов для грантов, тендеров и партнерств.
10. Формировать регулярные executive-дайджесты из разрозненных материалов.

### Штатные кейсы (задача / ожидаемые результаты)

1. **Задача:** В автономном режиме поручить агенту обновить оффер во всех материалах кампании. **Ожидаемые результаты:** правки в десятках файлов за минуты с проверяемым diff.
2. **Задача:** В автономном режиме привести стиль 40 уроков к единому стандарту. **Ожидаемые результаты:** согласованный тон, структура и терминология по всему курсу.
3. **Задача:** В автономном режиме проанализировать 40-страничный договор и выделить риски. **Ожидаемые результаты:** структурный risk-brief и приоритизированные правки.
4. **Задача:** В автономном режиме сгруппировать 200 отзывов клиентов по повторяющимся болям. **Ожидаемые результаты:** карта болей и список приоритетов для действий.
5. **Задача:** В автономном режиме сверить 30 научных статей на противоречия методологии. **Ожидаемые результаты:** таблица расхождений и рабочие гипотезы.
6. **Задача:** В автономном режиме сформировать пакет клиентских документов из шаблонов. **Ожидаемые результаты:** готовый комплект без ручной склейки.
7. **Задача:** В автономном режиме классифицировать входящую почту по срочности и типу ответа. **Ожидаемые результаты:** очередь обработки с приоритетами и черновиками ответов.
8. **Задача:** В автономном режиме нормализовать 20 таблиц с разными форматами метрик. **Ожидаемые результаты:** единая структура данных для анализа.
9. **Задача:** В автономном режиме подготовить персонализированные письма для 8 сегментов аудитории. **Ожидаемые результаты:** адаптированные версии с единым позиционированием.
10. **Задача:** В автономном режиме собирать еженедельный дайджест для руководителя из 15 источников. **Ожидаемые результаты:** краткий отчет с фактами, рисками и рекомендациями.

### Стек уровня

- IDE + Git/GitHub + AI-агент (Cline/Claude Code и аналоги).
- Diff-first контроль изменений.
- Правила постановки задач: ТЗ, критерии приемки, чеклисты.

### Подходы уровня

1. Формулировать задачу как «сделай», а не «посоветуй».
2. Описывать результат через критерии приемки и контрольные точки.
3. Переводить диалог в автономное исполнение: меньше микрошагов, больше проверки.
4. Проверять результат через diff, чеклисты и автоматические правила.

### Параметры эскалации (триггеры перехода)

- Появилась потребность в интерактивном интерфейсе, а не только в документах/таблицах.
- Нужны пользовательские сценарии (форма, личный кабинет, калькулятор, мини-сервис).
- Нужно быстро собрать рабочий прототип приложения без глубокой инженерии.

### Миссии-препятствия (требуют следующий уровень)

## L8. Финальный уровень: агентные системы и следующий курс

Это уровень «что дальше»: объясняем, почему ассистенты перестают хватать и почему агентные системы дают кратный рывок.

### Итог уровня (для преподавателя)

Инструменты: автоматизации + агентные фреймворки + контекстная память (на уровне концепции).  
Что меняется/результат: студент понимает логику агентных систем и готов к следующему курсу по агентам.  
Примеры задач (из кейсов уровня): агент отправляет результат в CRM, агент выбирает источник ответа, агент координирует выпуск книги.

### Что показывать на уровне (преподавателю)

Что показать: различие «ассистент vs агент», роль процесса, агентные роли и цепочки.  
Инструменты: Make/Zapier/n8n, концепции агентных фреймворков и памяти.  
Навык: проектировать процесс под агентный режим и понимать, что можно автоматизировать без ИИ.  
Концепция: агент = процесс + роли + память, а не «умный чат».

### Штатные задачи уровня (миссии развития)

1. Описать бизнес-процесс так, чтобы его можно было автоматизировать или поручить агенту.
2. Разделить процесс на задачи: что автоматизируется без ИИ, а где нужен агент.
3. Настроить агенту роль «исполнителя», а не «советника».
4. Встроить контекстную память: предпочтения, стиль, правила, повторяемые шаблоны.
5. Запустить агентный сценарий, который сам выбирает действия по логике.
6. Научить агента адаптироваться по обратной связи и корректировать поведение.
7. Делегировать агенту исследование и подготовку решений под неопределенность.
8. Организовать цепочку из нескольких агентов с распределением ролей.
9. Перевести контентный продукт в агентный сервис (регулярное выполнение).
10. Подготовить карту перехода в профессиональный курс по агентам.

### Штатные кейсы (задача / ожидаемые результаты)

1. **Задача:** Агент сам получает текст и отправляет его клиенту/в CRM. **Ожидаемые результаты:** цепочка «получил -> сделал -> отправил» без ручного шага.
2. **Задача:** Агент преобразует таблицу в отчет и публикует результат. **Ожидаемые результаты:** готовый отчет без ручного копипаста.
3. **Задача:** Агент сам выбирает, где искать ответ: интернет, ваши документы или база знаний. **Ожидаемые результаты:** корректный выбор источника и объяснение решения.
4. **Задача:** Агент ведет подготовку к событию: материалы, люди, план. **Ожидаемые результаты:** чек-лист и готовый пакет документов.
5. **Задача:** Агент помнит ваш стиль и готовит презентации «как вы любите». **Ожидаемые результаты:** согласованная структура и тон без правок «вручную».
6. **Задача:** Агент учится на вашей обратной связи и корректирует рекомендации. **Ожидаемые результаты:** заметное улучшение качества с каждой итерацией.
7. **Задача:** Агент исследует проблему проекта и предлагает варианты решения. **Ожидаемые результаты:** анализ причин + список действий с аргументами.
8. **Задача:** Агент координирует выпуск книги (редактор, верстка, обложка). **Ожидаемые результаты:** управляемый проект с дедлайнами и статусами.
9. **Задача:** Агент ведет семейный/корпоративный архив через «команду» подагентов. **Ожидаемые результаты:** системная поддержка архива без ручной рутины.
10. **Задача:** Агент-помощник по развитию карьеры предлагает персональный план перехода. **Ожидаемые результаты:** план обучения, шаги, варианты.

### Стек уровня

- Инструменты автоматизации процессов (Make/Zapier/n8n).
- Агентные фреймворки, сценарная логика, маршрутизация задач.
- Контекстная память и базы знаний (RAG/knowledge graph на уровне концепции).
- Оркестрация «агент -> агент» и роль человека как менеджера.

### Подходы уровня

1. Сначала процесс и критерии результата, потом агент.
2. Разделять «что можно автоматизировать без ИИ» и «где нужен ИИ».
3. Двигаться от простых ролей к адаптивным агентам.
4. Постоянный цикл: запуск -> обратная связь -> корректировка.

### Параметры эскалации (триггеры перехода)

- Агентная система должна работать в production с SLA и безопасностью.
- Нужны сложные бизнес-правила, тестируемость и мониторинг.
- Требуется масштабирование на большое число клиентов/процессов.

### Миссии-препятствия (требуют следующий уровень)

3. **Построить многоуровневую агентную организацию.** -> Нужна оркестрация, eval-контуры и governance. -> **L9+**.
