# Глава 6. Как перейти от удачных разовых запросов к повторяемым рабочим процессам в Claude Code

**Для кого эта глава**

Эта глава написана для человека, который:
- уже получил несколько хороших результатов от Claude Code;
- заметил, что каждый раз работа начинается почти с нуля;
- хочет перестать зависеть от случайного удачного запроса;
- хочет сделать работу с агентом более устойчивой и повторяемой.

## Зачем нужна эта глава

Разовая удача — это еще не система.

Если один раз получилось хорошо, это приятно.
Но зрелая работа начинается тогда, когда хороший результат можно повторять.

Именно здесь появляется следующая единица зрелости:
не просто хороший запрос,
а repeatable workflow.

## Что вы сможете после этой главы

После этой главы вы сможете:
- отличать случайный удачный результат от повторяемого процесса;
- использовать простой базовый цикл работы с Claude Code;
- собирать свои шаблоны и custom commands;
- понимать, когда процесс уже можно запускать по расписанию или через автоматизацию;
- не автоматизировать хаос раньше времени.

## Главная мысль главы

Сильный пользователь агента — это не тот, кто один раз придумал красивый prompt.
Сильный пользователь — это тот, кто умеет много раз повторить хороший результат по понятной схеме.

Если перевести это на простой язык:
workflow — это последовательность шагов, которая снова и снова приводит к приемлемому результату.

## Что такое workflow простыми словами

Workflow — это не одна реплика, а устойчивый порядок шагов.

Например:
1. попросить обзор;
2. попросить план;
3. дать маленькую исполнительную задачу;
4. проверить результат;
5. зафиксировать полезный итог.

Если вы повторяете это снова и снова, у вас уже появляется workflow.

## Как понять, что у вас пока нет workflow

### Признак 1. Каждый раз начинаете с чистого листа

### Признак 2. Хороший результат нельзя внятно повторить

### Признак 3. Вы не можете объяснить свой процесс другому человеку в 5-7 шагах

### Признак 4. Проверка каждый раз происходит по-разному

Если эти признаки есть, значит пока у вас не система, а цепочка случайных удач.

## Самый полезный базовый цикл

Для нетехнического пользователя очень хорошо работает такая лестница:

```text
Обзор -> План -> Действие -> Проверка -> Фиксация
```

### Шаг 1. Обзор

Примеры запросов:
- "Объясни, что здесь происходит";
- "Какие здесь главные файлы";
- "Где новичок запутается".

### Шаг 2. План

Примеры:
- "Составь план улучшения";
- "Покажи 5 шагов, но пока ничего не меняй".

### Шаг 3. Действие

Пример:
- "Сделай только первый шаг из этого плана".

### Шаг 4. Проверка

После этого вы смотрите:
- что изменилось;
- не ушел ли смысл;
- не расширилась ли область задачи.

### Шаг 5. Фиксация

Фиксация — это момент, когда вы:
- принимаете результат;
- сохраняете полезный шаблон;
- обновляете связанные артефакты;
- понимаете, что этот цикл можно повторить.

## Какие workflows полезны почти всем

### Workflow 1. Разобраться в новой папке

```text
1. Обзор структуры
2. Выделение главных файлов
3. Поиск слабых мест
4. План первого улучшения
5. Одна маленькая правка
6. Проверка результата
```

### Workflow 2. Проверка новой главы

```text
1. Прочитать главу
2. Найти места, где новичок споткнется
3. Составить план улучшений
4. Исправить один раздел
5. Проверить diff
6. Обновить оглавление
```

### Workflow 3. Регулярная редакторская проверка

```text
1. Просмотреть новые главы
2. Найти слишком сложные места
3. Найти недообъясненные интерфейсные шаги
4. Предложить точечные правки
5. Применить одну группу правок
6. Проверить итог
```

## Как сохранять workflow, чтобы он не терялся

Если хороший цикл повторяется, его надо выносить из головы во внешний артефакт.

Это можно делать так:
- в `CLAUDE.md`, если это постоянное правило;
- в отдельном Markdown-файле, если это шаблон процесса;
- в custom command, если процесс хочется запускать удобнее;
- позже — в scheduled task или automation.

## Что такое custom commands

По документации Anthropic Claude Code поддерживает custom commands.

Простыми словами:
это собственные команды, которыми вы упаковываете повторяемый процесс.

Например, вместо того чтобы каждый раз писать длинный запрос, у вас может быть команда вроде:
- `/review-chapter`
- `/prepare-outline`
- `/editorial-pass`

Это полезно потому, что:
- меньше ручной рутины;
- меньше забытых шагов;
- легче повторять одинаковый порядок.

## Когда custom command уже уместна

Custom command имеет смысл создавать, если процесс:
- повторялся уже несколько раз;
- имеет понятный вход;
- имеет понятный выход;
- почти всегда идет по одной и той же схеме.

## Что такое задачи по расписанию

В экосистеме Claude Code есть сценарии, где повторяющиеся промпты можно запускать по расписанию.
Также Desktop app поддерживает recurring tasks.

Простыми словами:
это фоновая регулярная работа, которая стартует не вручную, а по расписанию.

Но тут действует очень важное правило:

### Сначала стабилизируйте процесс вручную

И только потом переводите его в регулярный режим.

Иначе вы будете просто автоматически повторять ошибки.

## Когда процесс уже можно автоматизировать

Это безопасно делать, если:
- вход процесса стабилен;
- выход понятен и проверяем;
- есть точка качества;
- процесс уже проходил вручную много раз по одной схеме.

Например, хороший кандидат:
- раз в неделю проверять новые главы на места, где новичок может запутаться.

## Что пока лучше оставить ручным

Оставляйте ручным то, что:
- требует стратегического выбора;
- меняет структуру проекта;
- связано с высоким риском;
- требует человеческого вкуса и окончательного решения.

## Как собрать свой первый repeatable workflow

### Шаг 1. Возьмите задачу, которая повторялась минимум три раза

### Шаг 2. Запишите реальные шаги

Не идеальные, а реальные.

### Шаг 3. Уберите все лишнее

### Шаг 4. Зафиксируйте вход и выход

### Шаг 5. Отметьте точку проверки

### Шаг 6. Сохраните workflow в явном виде

Например:

```md
# Workflow: Проверка новой главы

1. Прочитать главу
2. Найти сложные места
3. Составить план улучшений
4. Переписать один раздел
5. Проверить diff
6. Обновить оглавление
```

## Частые ошибки

### Ошибка 1. Думать, что workflow — это что-то слишком сложное

Нет. Иногда это просто 5 устойчивых шагов.

### Ошибка 2. Автоматизировать нестабильный процесс

### Ошибка 3. Пытаться превратить творческое решение в механический конвейер

### Ошибка 4. Не сохранять удачные шаблоны

### Ошибка 5. Делать процесс без точки проверки

## Практическое упражнение

### Цель упражнения

Вытащить один повторяющийся процесс из головы и оформить его как рабочий шаблон.

### Что сделать

1. Возьмите задачу, которую вы уже делали с Claude Code минимум три раза.
2. Запишите 5-7 шагов, которые вы реально проходите.
3. Отдельно подпишите:
- вход;
- выход;
- точку проверки;
- то, что пока нельзя автоматизировать.
4. Сохраните этот шаблон как отдельную заметку или рабочий Markdown-файл.

## Как эта глава связана с более глубокой логикой курса

До этого вы уже научились:
- знакомиться с агентом;
- ставить задачи;
- держать память проекта;
- управлять разрешениями;
- проверять первые изменения.

Теперь все это собирается в следующую единицу зрелости:
не отдельное действие, а повторяемый цикл.

Именно из таких циклов потом вырастают:
- безопасная автономность;
- custom commands;
- регулярные запуски;
- более взрослая агентная среда.

## Словарь терминов

**Workflow**
Повторяемая последовательность шагов для решения похожих задач.

**Повторяемость**
Способность процесса снова и снова давать предсказуемый результат.

**Custom command**
Собственная команда, в которую упакован повторяемый процесс.

**Recurring task**
Повторяющаяся задача, которая запускается по расписанию.

**Точка проверки**
Момент процесса, где человек проверяет качество и границы результата.
