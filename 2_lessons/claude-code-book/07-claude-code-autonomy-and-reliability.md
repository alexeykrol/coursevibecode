# Глава 7. Как повышать автономность и надежность в Claude Code без потери контроля

**Для кого эта глава**

Эта глава написана для человека, который:
- уже умеет работать с Claude Code на понятных циклах;
- хочет, чтобы агент делал больше самостоятельно;
- не хочет вместе с автономностью получить хаос;
- хочет понять, как поднимать свободу постепенно и безопасно.

## Зачем нужна эта глава

Автономность очень легко понять неправильно.

Новичку кажется, что автономность — это просто больше свободы.
Но если свобода выросла, а надежность не выросла, это не зрелость.
Это просто риск.

Правильный рост автономности выглядит так:
- агент делает больше сам;
- вы вмешиваетесь реже;
- качество не падает;
- количество откатов не растет;
- эскалация происходит только там, где она действительно нужна.

## Что вы сможете после этой главы

После этой главы вы сможете:
- отличать автономность от бесконтрольной свободы;
- видеть ступени роста самостоятельности Claude Code;
- понимать роль permission modes, hooks, skills и subagents в надежной работе;
- решать, какие задачи уже можно делегировать дальше, а какие еще рано;
- измерять не только скорость, но и реальную надежность.

## Главная мысль главы

Цель не в том, чтобы агент делал вообще все сам.
Цель в том, чтобы он делал больше сам без потери качества и контроля.

Если после роста автономности вам приходится больше:
- откатывать;
- перепроверять;
- объяснять заново;
- чинить последствия;

значит, автономность была поднята слишком рано.

## Пять уровней зрелости автономности

### Уровень 1. Агент-объяснитель

Он:
- читает;
- объясняет;
- показывает структуру;
- отвечает на вопросы.

Почти ничего не меняет.

### Уровень 2. Агент-планировщик

Он:
- анализирует материал;
- предлагает план;
- помогает превратить идею в задачу.

### Уровень 3. Агент-исполнитель в узкой зоне

Он:
- меняет один файл;
- делает локальную правку;
- работает в ограниченной области.

Для большинства новичков это первый по-настоящему рабочий уровень.

### Уровень 4. Агент устойчивого процесса

Он:
- проходит известный workflow;
- действует внутри repeatable cycle;
- использует custom commands, память и проектные правила.

### Уровень 5. Агент делегирования и координации

Он:
- подключает subagents;
- распределяет отдельные подзадачи;
- собирает результат;
- эскалирует реальные тупики.

Это уже мост к следующему уровню — к агентной организации.

## Почему нельзя перепрыгивать уровни

Если вы еще не умеете:
- ставить ясные задачи;
- держать память проекта;
- ограничивать доступ;
- проверять результат;

то рост автономности почти всегда превращается в рост хаоса.

## Что создает надежность

Надежность собирается из нескольких опор.

### Опора 1. Ясная задача

Чем выше свобода агента, тем важнее:
- четкая цель;
- понятный материал;
- ограничения;
- критерий готовности.

### Опора 2. Память проекта

Если `CLAUDE.md` и память проекта не настроены, агент каждый раз стартует почти заново.

### Опора 3. Permission modes и границы доступа

Если разжать режим слишком рано, надежность падает.

### Опора 4. Проверка результата

Автономность без проверки быстро превращается в слепое доверие.

### Опора 5. Повторяемые процессы

Когда работа идет по стабильному workflow, надежность растет.

## Какую роль играют permission modes

### `Ask permissions`

Подходит для первых рабочих циклов и рискованных участков.

### `Plan mode`

Полезен, когда задача большая или неясная.
Он помогает не перепутать автономность с ранним хаотичным исполнением.

### `Auto accept edits`

Хорош для локальных, понятных и часто повторяемых изменений, если вы уже уверены в масштабе задач.

### `Bypass permissions`

Это не "продвинутый режим для всех".
Это режим для очень контролируемой и изолированной среды.

## Какую роль играют hooks

Hooks нужны не для красоты.
Они нужны, когда вы хотите встроить в процесс автоматические предохранители.

Например:
- не завершать работу без проверки;
- запускать форматирование после изменения;
- удерживать требования качества;
- не давать опасным действиям проходить тихо.

Для новичка это не первый инструмент.
Но позже hooks становятся важной частью надежности.

## Какую роль играют skills

По документации skills в Claude Code нужны для повторяемых рабочих пакетов.

Простыми словами:
skill — это упакованный рабочий навык.

Например:
- проверка главы на сложные места;
- редакторский проход;
- формат проверки PR.

Skill повышает надежность, потому что уменьшает число импровизаций.

## Какую роль играют subagents

По документации Claude Code поддерживает subagents.

Subagent — это специализированный помощник для отдельного типа работы.

Например:
- исследователь;
- ревьюер;
- планировщик;
- редактор.

Subagents полезны, когда уже понятно:
- какие роли нужны;
- какой формат результата ожидается;
- кто собирает итог.

Если этого нет, subagents только усложнят систему.

## Как повышать автономность безопасно

### Шаг 1. Возьмите процесс, который уже стал повторяемым

### Шаг 2. Убедитесь, что у процесса есть стабильный вход и выход

### Шаг 3. Убедитесь, что есть точка проверки

### Шаг 4. Увеличивайте свободу на один шаг, а не на пять

Например:
- вчера агент только строил план;
- сегодня пусть делает один проверяемый шаг;
- завтра пусть проходит весь малорисковый workflow.

### Шаг 5. Смотрите не только на скорость, но и на количество возвратов

Если возвратов и исправлений стало больше, свобода была расширена слишком рано.

## Как понять, что автономность выросла правильно

### Признак 1. Вы реже вмешиваетесь по мелочам

### Признак 2. Результат стал более ровным по качеству

### Признак 3. Агент сам проходит обычную неопределенность

### Признак 4. Эскалация стала реже, но содержательнее

## Когда автономность надо сужать

Сужайте ее, если:
- задачи снова стали расплывчатыми;
- проект быстро меняется;
- растет число побочных ошибок;
- Claude часто затрагивает лишние области;
- вы потеряли ощущение контроля.

Это не шаг назад.
Это возврат к управляемому состоянию.

## Частые ошибки

### Ошибка 1. Считать автономность просто количеством разрешений

### Ошибка 2. Просить автономность там, где еще нет стабильного workflow

### Ошибка 3. Подключать subagents слишком рано

### Ошибка 4. Не измерять качество после роста автономности

### Ошибка 5. Считать эскалацию слабостью

Нет. Хорошая система должна уметь эскалировать настоящие тупики.

## Практическое упражнение

### Цель упражнения

Понять, какой уровень автономности для одной вашей задачи уже безопасен.

### Что сделать

1. Возьмите одну повторяющуюся задачу.
2. Ответьте на четыре вопроса:
- у задачи есть стабильный вход;
- у задачи есть проверяемый выход;
- у задачи есть точка проверки;
- у задачи есть проектные правила в памяти или `CLAUDE.md`.
3. Если хотя бы на два вопроса ответ `нет`, не поднимайте автономность выше уровня локального исполнителя.
4. Если почти везде ответ `да`, можно думать о более самостоятельном прохождении workflow.

## Как эта глава связана с более глубокой логикой курса

В прошлой главе вы собрали repeatable workflows.
Теперь добавляется следующая ось:
не просто повторяемость, а безопасная самостоятельность.

Это уже прямой мост к следующему уровню:
`Agent Operating System`.

Потому что в какой-то момент становится ясно:
автономность держится не на одном хорошем запросе,
а на всей среде, в которой агент живет.

## Словарь терминов

**Автономность**
Способность агента двигаться вперед без постоянного ручного сопровождения.

**Надежность**
Способность системы стабильно давать приемлемый результат без лишнего хаоса.

**Управляемая агентность**
Способность агента проходить обычную рабочую неопределенность внутри заданных правил, качества и безопасности.

**Skill**
Упакованный рабочий навык для повторяемого типа задач.

**Subagent**
Специализированный помощник для отдельного типа работы.

**Эскалация**
Момент, когда задача или тупик поднимается человеку или более высокому уровню управления.
