# Глава 8. Что такое Agent Operating System вокруг Claude Code

**Для кого эта глава**

Эта глава написана для человека, который:
- уже понял, что хороший результат зависит не только от удачного запроса;
- видит, что без устойчивой среды работа с агентом все еще слишком хрупкая;
- хочет перейти от набора приемов к целостной системе;
- готов думать не только о задаче, но и о среде, в которой агент живет каждый день.

## Зачем нужна эта глава

До этого момента вы уже научились:
- знакомиться с агентом без хаоса;
- ставить задачи;
- строить память проекта;
- управлять permission modes;
- проверять первые изменения;
- собирать repeatable workflows;
- постепенно повышать автономность.

Теперь появляется новый вопрос.

Не такой:
- "что попросить Claude сейчас?"

А такой:
- "как вообще должен быть устроен проект, чтобы Claude Code работал в нем устойчиво каждый день?"

Вот здесь и появляется следующий уровень:
`Agent Operating System`.

## Что вы сможете после этой главы

После этой главы вы сможете:
- понять, что `Agent Operating System` — это не отдельная программа, а слой организации среды;
- отличать хороший prompt от хорошей рабочей среды;
- увидеть, из каких частей такая среда собирается в Claude Code;
- понять разницу между `runtime baseline`, `project overlay`, `preflight` и конкретным запуском;
- собрать минимальный взрослый набор вокруг Claude Code.

## Главная мысль главы

`Agent Operating System` — это не еще один умный режим.
Это устойчивая среда для работы агента.

Если сказать совсем просто:
это не сам работник,
а весь его рабочий кабинет.

В этом кабинете заранее определено:
- какие инструкции действуют;
- какие настройки включены;
- где границы доступа;
- какие проверки обязательны;
- какие skills и subagents доступны;
- как проходит обычная работа;
- когда агент должен эскалировать задачу человеку.

## Почему одного хорошего запроса уже недостаточно

Один хороший запрос решает одну задачу.

Но он не отвечает на более глубокие вопросы:
- как Claude должен работать в этом проекте вообще;
- что для него считается нормальным поведением;
- где у него границы;
- где хранятся правила;
- что он должен помнить;
- какие workflows должны повторяться одинаково.

Именно поэтому зрелость идет так:
- сначала хаотическое общение;
- потом хорошие задания;
- потом устойчивая среда.

## Из чего такой уровень собирается в Claude Code

В Claude Code этот уровень проявляется не одной кнопкой, а набором слоев.

### 1. `CLAUDE.md`

Это постоянные инструкции проекта.

### 2. `settings.json`

Это слой конфигурации и политики поведения.

### 3. Permission modes и permissions policy

Это слой реальной автономности и границ.

### 4. `.claude/rules/`

Это более узкие тематические правила для отдельных частей проекта.

### 5. Hooks

Это жесткие предохранители и автоматические quality gates.

### 6. Skills

Это повторяемые рабочие пакеты.

### 7. Subagents

Это слой специализации и распределения ролей.

### 8. MCP

Это слой внешних инструментов и подключений.

### 9. Auto memory

Это слой накопленной практической памяти о проекте.

### 10. Custom commands и recurring tasks

Это слой повторяемых действий и регулярных запусков.

Если сказать еще проще:
`Agent Operating System` вокруг Claude Code — это правильно собранная система инструкций, настроек, памяти, границ, специализации и процессов.

## Очень важная идея: среда важнее одного запуска

Если среда не собрана, каждый запуск будет:
- частично случайным;
- частично ручным;
- зависеть от того, вспомнили ли вы нужные правила именно сегодня.

Если среда собрана, отдельный запуск становится намного устойчивее.

То есть зрелость растет не потому, что вы сегодня особенно удачно написали запрос.
Она растет потому, что Claude стартует не из пустоты, а из правильной среды.

## Что такое `runtime baseline`

`Runtime baseline` — это базовый установочный комплект среды.

Простыми словами:
это минимальный стандарт, который вы хотите иметь до начала любой реальной работы.

Например, в baseline могут входить:
- базовый `CLAUDE.md`;
- базовый `settings.json`;
- стартовый профиль permissions;
- базовые hooks;
- набор базовых skills;
- минимальные правила эскалации.

Если baseline нет, агент каждый раз стартует почти как импровизация.

## Что такое `project overlay`

После baseline идет следующий слой:
`project overlay`.

Это уже не общие правила вообще,
а то, что относится именно к конкретному проекту.

Например:
- проект рассчитан на нетехническую аудиторию;
- все интерфейсные действия надо объяснять буквально;
- одна большая глава = один файл;
- после новой главы надо обновлять оглавление.

Иначе говоря:
- baseline = общий комплект среды;
- overlay = надстройка под конкретный проект.

## Что такое `preflight`

`Preflight` — это короткая проверка перед запуском задачи.

Простыми словами:
это проверка, готова ли среда к нормальной работе.

Например:
- открыт ли правильный проект;
- подхватились ли `CLAUDE.md` и правила;
- стоит ли правильный permission mode;
- есть ли точка возврата;
- понятна ли текущая задача.

Очень многие ошибки происходят не из-за плохого агента, а из-за плохого старта.
Именно это preflight и лечит.

## Взрослая последовательность зрелости

Если собрать все в одну линию, получится так:

```text
Ad hoc -> ТЗ -> Agent Operating System -> Runtime baseline -> Project overlay -> Preflight -> Active run
```

По-человечески это значит:
1. сначала люди просто пробуют;
2. потом учатся ставить задачи;
3. потом строят среду работы;
4. потом делают базовый стандарт этой среды;
5. потом добавляют правила проекта;
6. потом проверяют стартовые условия;
7. и только после этого запускают реальную задачу.

## Как выглядит минимальный взрослый набор

Минимальный зрелый `Agent Operating System` вокруг Claude Code может состоять из пяти вещей.

### 1. Один нормальный `CLAUDE.md`

### 2. Понятный профиль permissions

Например:
- `Ask permissions` на ранних этапах;
- более свободный режим только после появления надежного workflow.

### 3. Один-два repeatable workflows

Например:
- проверка новой главы;
- редакторский проход;
- обновление оглавления.

### 4. Явная точка проверки качества

Например:
- visual diff;
- backup-файл;
- ручная сверка изменения.

### 5. Правила эскалации

Например:
- агент проходит обычную рутину сам;
- человека зовет только при риске, развилке или конфликте целей.

## Когда понятно, что пора строить такой уровень

### Симптом 1. Вы постоянно повторяете одни и те же инструкции

### Симптом 2. Качество плавает от запуска к запуску

### Симптом 3. Страшно разжимать автономность

### Симптом 4. Проект уже длинный, а работа с агентом все еще держится только на текущем разговоре

### Симптом 5. Вы хотите, чтобы агент был частью системы, а не случайным собеседником

## Как начать собирать свой Agent Operating System без перегруза

### Шаг 1. Зафиксируйте постоянные правила

### Шаг 2. Зафиксируйте базовый профиль permissions

### Шаг 3. Зафиксируйте 1-2 repeatable workflows

### Шаг 4. Добавьте preflight-проверку

Например:
- та ли папка открыта;
- тот ли permission mode выбран;
- тот ли `CLAUDE.md` активен;
- есть ли точка возврата;
- понятна ли задача.

### Шаг 5. Зафиксируйте правила эскалации

## Что не надо делать

### Не надо думать, что Agent Operating System — это отдельная магическая программа

### Не надо пытаться собрать все сразу

Если вы в один день попытаетесь одновременно настроить:
- память;
- rules;
- hooks;
- skills;
- subagents;
- MCP;
- recurring tasks;

то почти наверняка утонете в сложности.

### Не надо строить среду без реальной практики

Среда должна расти из повторяющихся потребностей, а не из любви к архитектуре ради архитектуры.

## Частые ошибки

### Ошибка 1. Пытаться решить все одним prompt

### Ошибка 2. Писать слишком общий `CLAUDE.md`

### Ошибка 3. Смешивать baseline и проектные детали

### Ошибка 4. Пропускать preflight

### Ошибка 5. Повышать автономность без операционной среды

## Практическое упражнение

### Цель упражнения

Увидеть свой проект как среду для работы агента, а не как набор разовых разговоров.

### Что сделать

1. Возьмите текущий проект.
2. Отдельно выпишите:
- какие постоянные инструкции уже есть;
- какой профиль permissions реально используется;
- какие workflows уже устойчивы;
- где происходит проверка результата;
- когда агент должен эскалировать задачу человеку.
3. Затем задайте себе вопрос:

```text
Если завтра открыть новый разговор, насколько эта среда уже сама удержит качество без длинных повторных объяснений?
```

4. Если ответ слабый, значит ваш `Agent Operating System` еще только в зачатке.
5. Если ответ уверенный, значит среда начала складываться.

## Как эта глава связана с более глубокой логикой курса

На этом этапе происходит очень важный переход.

До этого вы учились хорошо работать с одним агентом в задачах и процессах.
Теперь вы увидели, что по-настоящему зрелая работа держится уже не только на агенте, а на всей среде вокруг него.

И это прямой мост к следующей главе.
Потому что как только среда одного агента становится устойчивой, следующим естественным вопросом становится:
- а что будет, если таких агентов станет несколько;
- как разделить роли;
- как координировать их работу;
- как строить уже не среду одного агента, а агентную организацию.

## Словарь терминов

**Agent Operating System**
Устойчивая среда для работы агента: инструкции, настройки, доступы, память, проверки, роли и повторяемые процессы.

**`Runtime baseline`**
Базовый установочный комплект среды до работы над конкретным проектом.

**`Project overlay`**
Проектная надстройка поверх baseline с правилами именно этого проекта.

**`Preflight`**
Проверка стартовых условий перед запуском конкретной задачи.

**`Active run`**
Конкретный рабочий запуск агента на реальной задаче.
