# Глава 3. Как устроены постоянные инструкции и память проекта в Claude Code

**Для кого эта глава**

Эта глава написана для человека, который:
- уже умеет давать Claude Code нормальные задачи;
- устал повторять одни и те же правила в каждом новом разговоре;
- хочет, чтобы агент лучше понимал конкретный проект;
- хочет перейти от разовых удачных ответов к более устойчивой работе.

## Зачем нужна эта глава

Если в каждой новой сессии вы снова объясняете:
- кто ваша аудитория;
- какой здесь стиль;
- что нельзя менять;
- где главные файлы;
- как выглядит хороший результат;

значит, у вас пока нет нормальной памяти проекта.

В Claude Code такая память строится не из магии, а из двух основных слоев:
- `CLAUDE.md` — правила, которые пишете вы;
- `auto memory` — заметки, которые Claude постепенно накапливает сам.

## Что вы сможете после этой главы

После этой главы вы сможете:
- понять разницу между `CLAUDE.md` и auto memory;
- выбрать, что писать вручную, а что можно оставить на автоматическое запоминание;
- создать свой первый рабочий `CLAUDE.md`;
- понять, где такие файлы могут лежать;
- проверить, действительно ли Claude Code их учитывает.

## Главная мысль главы

Хорошая память проекта — это не безграничная память обо всем.
Это правильно устроенный постоянный контекст.

Если сказать совсем просто:
- `CLAUDE.md` — это записка с правилами на стене рабочего кабинета;
- auto memory — это заметки, которые помощник делает себе сам по ходу работы.

Оба слоя полезны, но они нужны для разного.

## `CLAUDE.md` и auto memory: в чем разница

По официальной документации Claude Code оба механизма загружаются в начале каждой сессии, но работают по-разному.

### `CLAUDE.md`

Это то, что пишете вы.

Туда имеет смысл помещать:
- постоянные правила проекта;
- стиль работы;
- архитектурные решения;
- обязательные проверки;
- соглашения команды.

### Auto memory

Это то, что Claude записывает сам по ходу работы.

Туда обычно попадает:
- полезная команда сборки;
- найденная привычка проекта;
- повторяющееся замечание;
- вывод из прошлой работы, который пригодится потом.

Если коротко:
- `CLAUDE.md` = ваши явные правила;
- auto memory = накопленные рабочие заметки Claude.

## Что лучше класть в `CLAUDE.md`

### 1. Цель проекта

Например:
- это учебный курс для нетехнических людей;
- это книга по Claude Code;
- это база знаний для консультантов.

### 2. Аудиторию

Например:
- читатель не программист;
- интерфейсные шаги нужно объяснять буквально;
- технические термины нужно расшифровывать.

### 3. Правила построения материалов

Например:
- одна глава = один файл;
- конспект и полную главу не смешивать;
- после новой главы обновлять оглавление.

### 4. Постоянные ограничения

Например:
- не уходить в разработческий жаргон;
- не трогать другие файлы без просьбы;
- не заставлять читателя бегать по внешним ссылкам ради понимания основного текста.

## Что лучше не класть в `CLAUDE.md`

### Не кладите туда секреты

Например:
- пароли;
- ключи;
- токены;
- конфиденциальные данные, которые не нужны агенту.

### Не превращайте его в мусорный склад

Плохо, когда файл содержит:
- все мысли за много месяцев;
- абстрактную философию без практической пользы;
- старые и противоречивые правила;
- случайные временные пожелания.

По документации Anthropic для `CLAUDE.md` лучше работает короткий, конкретный и хорошо структурированный текст.

## Где может лежать `CLAUDE.md`

Официальная документация описывает несколько уровней.

### 1. Пользовательский уровень

Путь:

```text
~/.claude/CLAUDE.md
```

Это ваши общие личные предпочтения для всех проектов.

### 2. Уровень проекта

Пути:

```text
./CLAUDE.md
```

или

```text
./.claude/CLAUDE.md
```

Это главный проектный слой.
Он обычно хранится вместе с проектом и подходит для командных правил.

### 3. Более узкие правила

Для больших проектов документация рекомендует использовать каталог:

```text
.claude/rules/
```

Туда выносятся отдельные тематические правила.
Например:
- стиль;
- тесты;
- безопасность;
- правила для конкретной зоны проекта.

## Как понять эти уровни на бытовом примере

Представьте три таблички.

Первая висит у вас дома:
- как вы в целом любите работать.

Вторая висит в конкретном офисе проекта:
- как принято работать именно здесь.

Третья висит в определенной комнате:
- как работать именно с этой частью проекта.

С Claude Code логика та же:
- есть личные правила;
- есть проектные;
- есть более узкие локальные правила.

## Как создается проектный `CLAUDE.md`

Есть два пути.

### Путь 1. Создать файл вручную

Это самый понятный путь для нетехнического человека.

1. Откройте папку проекта.
2. В корне проекта создайте файл:

```text
CLAUDE.md
```

3. Откройте его.
4. Впишите туда простые и постоянные правила проекта.
5. Сохраните файл.

### Путь 2. Использовать `/init`

По официальной документации команда `/init` умеет создать стартовый `CLAUDE.md` автоматически.
Claude анализирует проект и предлагает начальный файл с командами, соглашениями и замеченными шаблонами.

Для новичка лучший подход такой:
- можно использовать `/init`, если хочется стартовой заготовки;
- но потом обязательно читать и править результат вручную.

Не нужно думать, что сгенерированный файл автоматически идеален.

## Как вызвать `/init`

### В интерактивной сессии Claude Code

1. Откройте Claude Code внутри проекта.
2. Нажмите в поле ввода сообщения.
3. Введите:

```text
/init
```

4. Нажмите `Enter`.
5. Если Claude предложит создать или улучшить `CLAUDE.md`, внимательно прочитайте предложение до записи файлов.

Если вы не видите ожидаемого результата, не продолжайте автоматически.
Вернитесь к ручному созданию файла.

## Каким должен быть хороший `CLAUDE.md`

Ниже минимальный практический пример.

```md
# Инструкции проекта

## Что это за проект
- Это учебный курс по работе с агентами.
- Основная аудитория — люди без технической подготовки.

## Как писать материалы
- Пиши простым русским языком.
- Объясняй термины сразу.
- Если в тексте есть интерфейсное действие, объясняй, где именно это находится.

## Как строить файлы
- Одна большая глава = один файл.
- После новой главы обновляй оглавление.
- Не смешивай конспект и полную главу в одном файле.
```

Такой файл уже полезен.
Он не обязан быть огромным, чтобы работать.

## Что такое auto memory

По документации Anthropic auto memory включена по умолчанию.

Простыми словами это означает:
Claude может сам сохранять полезные для будущей работы заметки.

Например:
- какая команда запускает проект;
- как обычно устроена сборка;
- какие решения уже выяснились в прошлых сессиях;
- какие ваши привычки повторяются снова и снова.

## Где хранится auto memory

По официальной документации память проекта хранится в каталоге вида:

```text
~/.claude/projects/<project>/memory/
```

Там есть:
- `MEMORY.md` — короткий входной индекс памяти;
- дополнительные тематические файлы, если Claude их создаст.

Важно:
- это память, локальная для вашей машины;
- она не становится автоматически общей для всей команды;
- она не заменяет `CLAUDE.md`.

## Как управлять auto memory

### Через команду `/memory`

По документации самый понятный путь для пользователя — команда:

```text
/memory
```

### Что делать пошагово

1. Откройте Claude Code в проекте.
2. Нажмите в поле ввода.
3. Введите:

```text
/memory
```

4. Нажмите `Enter`.
5. Должно открыться представление памяти или меню, где можно посмотреть сохраненные заметки и настройки auto memory.

Если вы видите переключатель auto memory, значит вы на правильном месте.

### Через настройки

По документации auto memory можно включать и выключать настройкой:

```json
{
  "autoMemoryEnabled": false
}
```

Но для новичка удобнее сначала работать через `/memory`, а не через ручное редактирование `settings.json`.

## Когда правило надо писать в `CLAUDE.md`, а когда можно доверить памяти

### Пишите в `CLAUDE.md`, если:
- правило должно действовать всегда;
- это норма проекта;
- это ограничение безопасности или качества;
- это важно для всех будущих сессий и всех участников.

### Оставляйте auto memory, если:
- это полезное наблюдение по ходу работы;
- это не правило, а накопленный практический вывод;
- это локальная подсказка, а не проектный закон.

## Как проверить, что Claude действительно видит ваши инструкции

Это обязательный шаг.

1. Сохраните `CLAUDE.md`.
2. Откройте новую сессию Claude Code или продолжите текущую.
3. Введите в поле сообщения:

```text
Скажи, какие постоянные инструкции проекта ты сейчас видишь и как ты их понимаешь.
```

4. Прочитайте ответ.

### Что должно произойти

Хороший ответ:
- перечислит ключевые правила;
- перескажет их своими словами;
- покажет, что агент понял аудиторию, стиль и ограничения.

### Что должно насторожить

Если ответ слишком общий, возможны такие причины:
- файл лежит не там;
- он назван неправильно;
- вы работаете не из той папки;
- есть более узкий или более высокий слой инструкций, который сейчас важнее.

## Частые ошибки

### Ошибка 1. Писать слишком абстрактно

Плохо:

```md
- Будь полезным.
- Пиши хорошо.
- Работай профессионально.
```

Лучше:

```md
- Пиши простым русским языком для нетехнического читателя.
- В конце большой главы добавляй словарь терминов.
- Если текст говорит пользователю что-то нажать, подробно объясняй где это находится.
```

### Ошибка 2. Класть временные пожелания в постоянные инструкции

Если правило нужно только для одной сессии, лучше держать его в запросе, а не превращать в вечный закон.

### Ошибка 3. Путать `CLAUDE.md` и auto memory

Они дополняют друг друга, но не заменяют.

### Ошибка 4. Не проверять, подхватились ли инструкции

Создать файл мало. Надо еще убедиться, что Claude его видит.

## Практическое упражнение

### Цель упражнения

Создать или улучшить проектную память в понятной и проверяемой форме.

### Что сделать

1. Откройте папку проекта.
2. Найдите или создайте `CLAUDE.md`.
3. Добавьте туда три вещи:
- цель проекта;
- аудиторию;
- одно постоянное правило по стилю или структуре.
4. Сохраните файл.
5. Затем откройте Claude Code и отправьте:

```text
Скажи, какие постоянные инструкции проекта ты сейчас видишь и как они повлияют на следующую задачу.
```

6. Если хотите, отдельно вызовите `/memory` и посмотрите, что хранится в памяти проекта.

## Как эта глава связана с более глубокой логикой курса

Во второй главе вы научились ставить ясные задачи.
Но если каждую задачу приходится собирать с нуля, у вас пока нет устойчивой среды.

Память проекта — это следующий шаг зрелости.
Она переводит работу:
- от разовых хороших формулировок;
- к постоянной проектной рамке.

Дальше к ней добавятся:
- настройки доступа;
- permission modes;
- hooks;
- repeatable workflows;
- автономность и надежность.

## Словарь терминов

**`CLAUDE.md`**
Markdown-файл с постоянными инструкциями для Claude Code.

**Auto memory**
Автоматически накапливаемые заметки, которые Claude сохраняет для будущих сессий.

**`/init`**
Команда, которая помогает создать стартовый `CLAUDE.md` автоматически.

**`/memory`**
Команда, которая позволяет посмотреть и управлять памятью проекта.

**`.claude/rules/`**
Папка, где можно хранить более узкие и модульные правила для проекта.
