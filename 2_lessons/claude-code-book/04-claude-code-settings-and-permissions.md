# Глава 4. Как управлять настройками, permission modes и безопасными границами в Claude Code

**Для кого эта глава**

Эта глава написана для человека, который:
- уже научился ставить Claude Code ясные задачи;
- хочет понять, какие настройки реально влияют на поведение агента;
- боится случайно дать слишком много свободы;
- хочет настроить среду так, чтобы агент был полезным, но не хаотичным.

## Зачем нужна эта глава

Очень многие новички путают две разные вещи:
- насколько агент умный;
- насколько агенту разрешено действовать.

Это не одно и то же.

Можно иметь сильную модель и при этом очень осторожный режим работы.
И можно, наоборот, разжать разрешения слишком рано и получить лишний риск.

Поэтому взрослая работа с Claude Code строится не только на хорошем запросе, но и на правильных настройках среды.

## Что вы сможете после этой главы

После этой главы вы сможете:
- понять, какие настройки реально важны на старте;
- различать модель, память, permission modes и hard limits;
- открывать настройки в Desktop app и проектных файлах;
- выбирать безопасный стартовый режим;
- понимать, когда достаточно `Ask permissions`, когда удобен `Auto accept edits`, а когда `Bypass permissions` уже опасен.

## Главная мысль главы

Безопасность работы с агентом начинается не с паники и не с тотального запрета.
Она начинается с правильных границ.

В Claude Code эти границы задаются несколькими слоями:
- permission mode в текущей сессии;
- `settings.json`;
- `CLAUDE.md` и проектные правила;
- при необходимости `hooks`, `rules`, `MCP` и более строгие ограничения.

Если перевести на простой язык:
- одна часть отвечает за то, как свободно Claude действует прямо сейчас;
- другая за постоянную политику;
- третья за проектные правила;
- четвертая за жесткие предохранители.

## Какие настройки реально важны новичку

На старте важны пять групп настроек:
1. permission mode;
2. память и инструкции;
3. `settings.json`;
4. доступ к опасным действиям;
5. дополнительные защитные слои вроде hooks.

Все остальное пока вторично.

## 1. Permission mode

Это самое важное место для начала.

По документации Desktop app у Claude Code есть такие режимы:
- `Ask permissions`;
- `Auto accept edits`;
- `Plan mode`;
- `Bypass permissions`.

Также документация отмечает, что режим `dontAsk` доступен только в CLI.

### `Ask permissions`

Это лучший стартовый режим для новичка.

Что он означает:
- Claude спрашивает перед изменением файлов;
- Claude спрашивает перед запуском команд;
- вы видите, что он собирается делать.

По документации Anthropic этот режим прямо рекомендован новым пользователям.

### `Auto accept edits`

Что он означает:
- правки файлов принимаются автоматически;
- но перед командами Claude все еще спрашивает.

Это уже более быстрый режим.
Он хорош, когда:
- вы уже доверяете локальным текстовым правкам;
- но не хотите отпускать команды без контроля.

### `Plan mode`

Что он означает:
- Claude анализирует код и строит план;
- не меняет файлы;
- не запускает команды.

Это отличный режим для:
- больших задач;
- неясных задач;
- первой разведки;
- согласования подхода.

### `Bypass permissions`

Что он означает:
- Claude работает без permission prompts;
- в документации этот режим прямо приравнивается к очень опасному флагу в CLI;
- его рекомендуют использовать только в sandboxed container или VM.

Для новичка это не стартовый режим.

### `dontAsk`

По документации этот режим есть только в CLI.
Он не появляется в Desktop app как обычный пользовательский режим.

## Простое правило для новичка

Если сомневаетесь:
1. начинайте с `Ask permissions`;
2. на сложных задачах используйте `Plan mode`;
3. к `Auto accept edits` переходите только когда уже уверены в масштабе локальных правок;
4. `Bypass permissions` не используйте в первые дни вообще.

## Где выбрать permission mode в Desktop app

По официальной документации selector режима находится рядом с кнопкой отправки сообщения.

### Пошагово

1. Откройте Claude Desktop.
2. Перейдите на вкладку `Code`.
3. Откройте нужную сессию.
4. Посмотрите в нижнюю часть окна.
Там находится поле ввода сообщения.
5. Рядом с ним найдите selector режима.
Обычно это выпадающее меню или кнопка с текущим названием режима.
6. Нажмите на него.
7. Выберите нужный режим, например `Ask permissions` или `Plan mode`.

Если вы не видите selector:
- попробуйте расширить окно;
- закройте и снова откройте сессию;
- убедитесь, что вы действительно в `Code`, а не в обычном чате.

## Что еще умеет Desktop app

По документации Desktop app также позволяет:
- визуально смотреть diff;
- запускать несколько сессий параллельно;
- работать с preview;
- запускать долгие задачи и облачные сессии.

Но для этой главы главное — режимы разрешений.

## 2. `settings.json`

Если permission mode — это режим текущей сессии,
то `settings.json` — это постоянная конфигурация среды.

По документации настройки могут жить на разных уровнях.

### Основная идея

- есть пользовательские настройки;
- есть проектные настройки;
- есть локальные настройки для конкретной машины.

Эта многоуровневая схема полезна потому, что не все надо тащить во все проекты одинаково.

## Где искать файл настроек

Для большинства пользователей главный путь выглядит так:

```text
~/.claude/settings.json
```

Для проектного уровня документация также описывает настройки внутри проекта.

Если говорить человеческим языком:
- общий стиль и постоянные личные правила живут в пользовательских настройках;
- проектные ограничения и специфика — ближе к самому проекту.

## Что в `settings.json` реально важно новичку

### Раздел `permissions`

По документации именно здесь задаются:
- режимы по умолчанию;
- allow/ask/deny;
- правила обхода или запрета опасных действий.

### Раздел памяти

Здесь может жить переключатель auto memory.

### Разделы, связанные с hooks, агентами и плагинами

Они важны позже, но в первый день необязательны.

## Что такое `allow`, `ask`, `deny`

Это очень полезная логика.

### `allow`

То, что Claude можно делать без дополнительной паузы.

### `ask`

То, что Claude должен согласовать с вами.

### `deny`

То, что Claude делать нельзя.

Если перевести это на простой язык:
- `allow` — можно;
- `ask` — спроси;
- `deny` — нельзя.

Именно так строится controlled autonomy.

## Почему нельзя бездумно жать `Allow`

Здесь есть один очень важный практический риск, который многие понимают слишком поздно.

Когда Claude Code предлагает команду и вы нажимаете `Allow`, локальная система разрешений может сохранить саму строку команды, чтобы в следующий раз не спрашивать вас снова.

Если в этой строке уже были:
- токен;
- пароль;
- приватный ключ;
- IP и логин сервера;
- строка подключения к базе;

то все это может оказаться в `.claude/settings.local.json`.

Это не тот случай, когда система "вдруг сошла с ума".
Это ожидаемое следствие хаотичного одобрения команд с чувствительными данными прямо inline.

## Реальный инцидент, который стоит запомнить

Такой риск уже проявлялся на практике:

- проблема была замечена случайно, когда человек просто открыл `.claude/settings.local.json` в IDE;
- внутри оказались не только настройки, но и полный приватный SSH-ключ, service role токен и серверные параметры;
- если бы файл случайно попал в git, это стало бы уже не локальной проблемой, а утечкой в историю репозитория.

Главный вывод очень простой:

не надо жать `Allow` на команды, где чувствительные данные стоят прямо в строке.

Сначала:
- вынесите секреты в отдельный локальный файл, например `.production-credentials`;
- перестройте команду так, чтобы в самой разрешаемой строке не было реальных токенов и ключей;
- только после этого решайте вопрос с разрешением.

## Какой подход безопаснее

Безопаснее так:

```text
"Bash(ssh:*)"
"Bash(mariadb:*)"
```

Опаснее так:

```text
"Bash(ssh -i ~/.ssh/prod_key -p 65002 user@1.2.3.4 ...)"
"Bash(curl -H 'Authorization: Bearer <token>' ...)"
```

В первом случае вы разрешаете тип действия.
Во втором — сохраняете конкретную строку с чувствительными данными.

## Что новичку стоит держать под контролем

На старте полезно особенно осторожно относиться к таким вещам:
- чтение чувствительных файлов;
- сетевые команды;
- cloud CLI;
- `git push`;
- все, что может быть необратимым или разрушительным.

## Что такое hooks

По документации `hooks` — это способ запускать команды до или после действий Claude Code.

Простыми словами:
это автоматические предохранители или автоматические помощники.

Например:
- после редактирования файла автоматически запускать форматирование;
- перед завершением задачи прогонять проверку;
- не давать завершить действие, если не выполнены условия качества.

## Что важно понять про hooks

Hook — это уже не просьба агенту "будь осторожен".
Это более жесткий слой поведения.

Для новичка достаточно понять:
- hooks нужны не в первый час;
- но они очень важны позже, когда вы хотите сделать процесс более надежным и менее зависящим от доброй воли агента.

## Когда hooks уже уместны

Их имеет смысл подключать, когда у вас уже есть:
- повторяемый workflow;
- понятная точка качества;
- ясное правило, которое должно работать всегда.

## Что новичку не надо трогать в первый день

Не надо в первый день:
- настраивать сложные hooks;
- включать `Bypass permissions`;
- строить сложную схему allow/ask/deny без понимания;
- подключать много внешних инструментов одновременно;
- менять сразу все настройки подряд.

## Безопасный стартовый сценарий

Вот практическая схема для начала.

### Шаг 1. Поставьте `Ask permissions`

### Шаг 2. На сложных задачах переключайтесь в `Plan mode`

### Шаг 3. Держите проектный `CLAUDE.md` коротким и ясным

### Шаг 4. Не разжимайте доступы, пока нет проверяемого workflow

### Шаг 5. Только после этого думайте про `Auto accept edits`, hooks и более высокую автономность

## Частые ошибки

### Ошибка 1. Путать permission mode с умом модели

Это разные вещи.

### Ошибка 2. Сразу включать самый свободный режим

Это не делает вас продвинутым пользователем.
Это просто раньше времени увеличивает риск.

### Ошибка 3. Бездумно жать `Allow` на команды с кредами inline

Так разрешения легко превращаются в хранилище секретов.

### Ошибка 4. Менять слишком много настроек одновременно

Если вы одновременно меняете:
- режим разрешений;
- память;
- project instructions;
- hooks;
- allow/ask/deny;

потом трудно понять, что именно повлияло на поведение.

### Ошибка 5. Считать hooks обязательными с первого дня

Нет.
Они полезны позже, когда процесс уже оформился.

## Практическое упражнение

### Цель упражнения

Понять, где находятся реальные рычаги безопасности и какой режим вам подходит сейчас.

### Что сделать

1. Откройте Desktop app или другую поверхность Claude Code.
2. Найдите selector режима рядом с полем ввода.
3. Переключитесь в `Ask permissions`.
4. Затем на одной большой задаче переключитесь в `Plan mode`.
5. Откройте `.claude/settings.local.json`, если он у вас уже есть, и проверьте, нет ли там командных строк с токенами, ключами, IP и логинами.
6. После этого ответьте себе на три вопроса:
- в каком режиме я лучше понимаю, что делает агент;
- где мне спокойнее всего;
- где начинается лишняя свобода.

Если вы уже готовы смотреть настройки глубже, откройте `settings.json` и просто прочитайте раздел `permissions`, не меняя пока ничего лишнего.

## Как эта глава связана с более глубокой логикой курса

В прошлой главе вы создали память проекта.
Теперь вы добавили следующий обязательный слой: управляемые границы.

Это уже очень близко к тому, что позже станет основой более зрелой среды:
- сначала ясные задачи;
- потом память;
- потом настройки и границы;
- потом repeatable workflows;
- потом безопасная автономность.

## Словарь терминов

**Permission mode**
Режим, который задает, когда Claude Code должен спрашивать разрешение перед действием.

**`Ask permissions`**
Режим для новых пользователей: Claude спрашивает перед изменением файлов и запуском команд.

**`Auto accept edits`**
Режим, в котором правки файлов принимаются автоматически, а команды все еще требуют согласования.

**`Plan mode`**
Режим анализа и планирования без изменений файлов и без запуска команд.

**`Bypass permissions`**
Очень свободный и рискованный режим без permission prompts. Документация советует использовать его только в сильно изолированной среде.

**`settings.json`**
Главный конфигурационный файл Claude Code.

**`settings.local.json`**
Локальный файл проектных разрешений и настроек. Его нельзя рассматривать как безопасное место для хранения секретов.

**`allow / ask / deny`**
Три типа политики действий: можно, спроси, нельзя.

**Hook**
Автоматическое правило или команда, которая срабатывает до или после действий Claude Code.
