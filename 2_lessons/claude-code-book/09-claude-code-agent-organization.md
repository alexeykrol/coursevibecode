# Глава 9. Как из работы с одним Claude Code вырастает агентная организация

**Для кого эта глава**

Эта глава написана для человека, который:
- уже понимает, как устроить устойчивую среду для одного агента;
- начинает упираться в предел одного универсального помощника;
- хочет понять, когда действительно нужны роли, делегирование и координация;
- хочет увидеть следующий уровень зрелости без фантазий и без шума.

## Зачем нужна эта глава

Один хороший агент может делать очень много.
Но в какой-то момент появляется естественный предел.

Например, одна длинная задача уже требует одновременно:
- исследования;
- планирования;
- написания;
- проверки качества;
- интеграции результата;
- координации нескольких кусков работы.

Тогда проблема уже не в том, что агент слабый.
Проблема в том, что весь объем ролей висит на одной единице.

Вот здесь и появляется следующий уровень:
не просто сильный агент,
а агентная организация.

## Что вы сможете после этой главы

После этой главы вы сможете:
- понять разницу между одним агентом, subagents и agent teams;
- увидеть, что организация живет не в модели, а в артефактах и правилах;
- понять, какие роли обычно появляются первыми;
- понимать, когда agent teams действительно полезны, а когда они только усложняют работу;
- собрать минимальную схему агентной организации вокруг Claude Code.

## Главная мысль главы

Агентная организация — это не просто несколько агентов, открытых одновременно.

Это система, в которой есть:
- роли;
- маршруты делегирования;
- общая память;
- статус задач;
- правила интеграции;
- правила эскалации;
- общая политика качества и безопасности.

Если сказать совсем просто:
это уже не один помощник,
а маленькая виртуальная организация труда.

## Почему несколько агентов — это еще не организация

Можно открыть:
- несколько сессий;
- несколько вкладок;
- несколько исполнителей.

Но это само по себе еще не дает организацию.

Организация появляется только тогда, когда есть:
- кто за что отвечает;
- как передаются задачи;
- где хранится общий backlog;
- кто собирает итог;
- когда вопрос поднимается человеку.

Без этого будет не организация, а несколько разрозненных голосов.

## Где организация живет на самом деле

Это один из самых важных моментов.

Организация живет не внутри "магически умной" модели.
Она живет во внешней системе:
- ролей;
- артефактов;
- backlog;
- статусов;
- правил делегирования;
- правил эскалации.

То есть организация живет в протоколах и артефактах, а не в фантазии о всесильном агенте.

## Какой переход здесь происходит

На предыдущем уровне единицей автономности был один агент внутри хорошей среды.

Теперь единицей автономности становится уже не один агент,
а система агентов, которая умеет:
- разложить работу;
- распределить роли;
- выполнять части параллельно;
- проверять и собирать результат;
- эскалировать только реальные тупики.

## Что такое subagents в этой картине

По документации Claude Code subagents — это специализированные помощники внутри одной сессии.

Их сильная сторона:
- они держат отдельный контекст;
- могут быть заточены под отдельный тип работы;
- полезны для узких и сфокусированных подзадач.

Примеры:
- исследователь;
- планировщик;
- ревьюер;
- редактор.

Subagents хорошо подходят, когда:
- вам нужен отдельный исполнитель на конкретную часть;
- не нужна сложная межагентная коммуникация;
- результат должен вернуться в основную сессию.

## Что такое agent teams

По актуальной документации Claude Code поддерживает `agent teams`.

Важно: на дату **18 марта 2026** документация помечает agent teams как **experimental** и **disabled by default**.
Чтобы их включить, нужно явно задать флаг `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` в настройках или окружении.

Простыми словами:
agent team — это не просто подзадача внутри одной сессии.
Это уже команда отдельных сессий Claude Code, где:
- один агент становится lead;
- другие становятся teammates;
- есть общая task list;
- агенты могут общаться друг с другом напрямую.

## Чем agent teams отличаются от subagents

По официальной документации разница такая.

### Subagents

- живут внутри одной сессии;
- имеют свой контекст;
- возвращают результат обратно в основную сессию;
- лучше подходят для быстрых и сфокусированных подзадач.

### Agent teams

- состоят из независимых сессий;
- имеют общую task list;
- могут координироваться друг с другом;
- лучше подходят для сложной параллельной работы, где важна коммуникация между участниками.

Если сказать очень просто:
- subagent — это помощник;
- agent team — это уже маленькая команда.

## Какие роли обычно появляются первыми

Не надо начинать с десяти ролей.
Почти всегда достаточно малого набора.

### Роль 1. Координатор

Он:
- понимает общую задачу;
- разбивает ее на части;
- следит за статусами;
- собирает итог;
- управляет эскалацией.

В agent teams этому очень близка роль lead.

### Роль 2. Исследователь

Он:
- собирает материалы;
- ищет слабые места;
- сравнивает варианты;
- подготавливает базу для решений.

### Роль 3. Исполнитель

Он:
- пишет;
- меняет файл;
- делает конкретный артефакт.

### Роль 4. Ревьюер

Он:
- проверяет качество;
- ищет ошибки;
- оценивает соответствие правилам;
- помогает не принять слабый результат за хороший.

### Роль 5. Интегратор

Иногда это тот же координатор.
Но на длинных работах это может быть отдельная роль.

Он:
- собирает куски вместе;
- снимает противоречия;
- проверяет, что части не конфликтуют.

## Как это выглядит вокруг Claude Code

В Claude Code для такой системы есть несколько строительных блоков:
- `CLAUDE.md` и project rules;
- skills;
- subagents;
- hooks;
- agent teams;
- общие артефакты вроде backlog и findings doc.

Но важно помнить:
эти инструменты сами по себе не создают организацию.
Организацию создает ваша архитектура ролей и артефактов.

## Самая простая минимальная организация

Если не усложнять, начать можно с трех ролей:
1. координатор;
2. исполнитель;
3. ревьюер.

### Как это работает

- координатор формулирует задачу и границы;
- исполнитель делает работу;
- ревьюер проверяет результат;
- координатор принимает итог или возвращает на доработку.

Даже такая схема уже взрослее, чем один агент, который сам себе и постановщик, и исполнитель, и контролер.

## Когда agent teams действительно нужны

По документации Anthropic agent teams полезны там, где параллельная работа дает реальную ценность.

Например:
- исследование и review с разными углами взгляда;
- новая функция, которую можно разрезать на независимые куски;
- баг с конкурирующими гипотезами;
- работа, где фронт, бэк и тесты можно вести параллельно.

## Когда agent teams пока не нужны

Agent teams добавляют coordination overhead и расходуют больше ресурсов.

Не усложняйте, если:
- задача маленькая;
- один агент спокойно справляется;
- работа последовательная, а не параллельная;
- несколько участников будут постоянно лезть в один и тот же файл;
- среда одного агента у вас еще неустойчива.

По документации для последовательных задач чаще лучше одна сессия или subagents, а не team.

## Что такое хорошее делегирование

Хорошее делегирование — это не фраза:
- "пусть кто-нибудь разберется".

Хорошее делегирование всегда включает:
- цель;
- границы;
- ожидаемый результат;
- формат отчета;
- правило эскалации.

## Что такое общая память организации

Если у каждого агента свой отдельный мир, организация долго не проживет.

Нужны общие артефакты.
Например:
- backlog;
- status-файл;
- findings doc;
- список решений;
- правила проекта.

По документации agent teams используют shared task list, а также локальные файлы конфигурации и задач.
Но поверх этого вам все равно полезно проектировать собственные организационные артефакты.

## Почему эскалация особенно важна на этом уровне

Хорошая организация не убирает эскалацию.
Она делает ее:
- редкой;
- содержательной;
- адресной.

По документации agent teams умеют эскалировать управление через lead и rules вроде plan approval.
Но человек все равно должен оставаться верхним уровнем для:
- конфликта целей;
- высокого риска;
- необратимых решений;
- случаев, где локально нет данных.

## Практическое упражнение

### Цель упражнения

Понять, можно ли одну вашу длинную задачу разложить на минимальную агентную организацию.

### Что сделать

1. Возьмите одну большую задачу, которая уже плохо помещается в одного агента.
2. Разрежьте ее минимум на три роли:
- кто исследует;
- кто делает;
- кто проверяет.
3. Отдельно ответьте:
- кто координирует;
- какой общий backlog или status-файл нужен;
- в каких случаях вопрос поднимается человеку.

## Как эта глава завершает текущую линию книги

Книга начиналась с очень простого вопроса:
как вообще подойти к Claude Code без страха и хаоса.

Потом постепенно выстроилась лестница:
- знакомство;
- постановка задач;
- память проекта;
- настройки и границы;
- первые изменения;
- repeatable workflows;
- автономность и надежность;
- Agent Operating System.

И теперь естественная вершина этой линии такая:
не просто один сильный агент,
а система разделения труда между агентами.

Это не значит, что всем нужен этот уровень прямо сейчас.
Но это значит, что теперь у вас есть полная карта зрелости:
от первого безопасного разговора до проектирования агентной организации.

## Словарь терминов

**Subagent**
Специализированный помощник внутри одной сессии Claude Code.

**Agent team**
Команда независимых сессий Claude Code с lead, teammates, общей task list и межагентной координацией.

**Lead**
Главная сессия команды, которая координирует работу teammates.

**Teammate**
Отдельный участник agent team со своим контекстом и своей подзадачей.

**Общая task list**
Совместный список задач, через который team координирует работу.

**Организационная агентность**
Способность системы агентов самой вести сложную работу, сохраняя согласованность, качество и разумные границы эскалации.
