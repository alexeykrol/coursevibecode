# Глава 5. Как делать первые правки, создавать точки возврата и проверять результат в Claude Code

**Для кого эта глава**

Эта глава написана для человека, который:
- уже понимает основы работы с Claude Code;
- готов перейти от анализа к первому реальному изменению;
- боится, что агент перепишет слишком много;
- хочет научиться принимать изменения осознанно, а не на доверии.

## Зачем нужна эта глава

У большинства новичков есть один и тот же страх:
"Пока агент объясняет — все спокойно. Но что будет, когда он начнет менять файлы?"

Это правильный страх.
Он защищает от беспечности.

Нужен не героизм и не отказ от правок.
Нужен правильный рабочий цикл:
- маленькая правка;
- ясная граница;
- точка возврата;
- проверка результата.

## Что вы сможете после этой главы

После этой главы вы сможете:
- выбрать безопасную первую правку;
- создать точку возврата до изменения;
- попросить Claude Code изменить только один маленький участок;
- посмотреть, что именно поменялось;
- принять только полезный результат или откатить лишнее.

## Главная мысль главы

Первая правка должна быть маленькой не потому, что вы слабый пользователь.
Она должна быть маленькой потому, что вы строите управляемый цикл.

Хороший цикл выглядит так:
1. выбрать маленькую задачу;
2. ограничить область;
3. сделать точку возврата;
4. выполнить изменение;
5. посмотреть diff;
6. принять или откатить.

## Что считать хорошей первой правкой

Первая правка должна быть:
- небольшой;
- понятной;
- легко проверяемой;
- не критичной для всей системы.

### Хорошие варианты

- переписать один короткий раздел;
- упростить формулировку;
- улучшить заголовок;
- сократить длинный абзац;
- поправить один список;
- добавить пояснение к термину.

### Плохие варианты

- переписать всю книгу;
- перестроить весь проект;
- сразу менять много файлов;
- одновременно менять структуру, стиль и смысл.

## Что такое точка возврата

Точка возврата — это способ вернуться к состоянию до правки.

Если говорить совсем просто:
это ваша страховка.

## Два практических способа сделать точку возврата

### Способ 1. Если проект под Git

Это самый удобный вариант.
Git умеет:
- помнить состояние файлов;
- показывать разницу;
- помогать откатывать лишние изменения.

### Способ 2. Если Git вам пока чужд

Перед первой правкой:
- сделайте копию файла;
- или дубликат папки;
- или backup-файл рядом.

Это не так красиво, как Git, но для первого шага абсолютно нормально.

## Самая безопасная схема первой правки

### Шаг 1. Выберите один файл

Один, а не пять.

Например:

```text
2_lessons/claude-code-book/01-claude-code-first-contact.md
```

### Шаг 2. Выберите один кусок внутри файла

Например:
- только вступление;
- только раздел с упражнением;
- только словарь;
- только один блок про интерфейсные действия.

### Шаг 3. Сделайте точку возврата

Если Git вам пока чужд, просто сохраните копию.

Например:

```text
01-claude-code-first-contact.backup.md
```

### Шаг 4. Дайте агенту узкую задачу

```text
Задача:
Перепиши раздел более простым языком.

Материал:
Работай только с разделом "Что вы сможете после этой главы" в файле 2_lessons/claude-code-book/01-claude-code-first-contact.md.

Ограничения:
Не меняй другие разделы. Не трогай другие файлы. Не меняй смысл.

Готово, когда:
Изменится только этот раздел, а формулировки станут проще и короче.
```

### Шаг 5. Проверьте, что изменилось

Не принимайте результат на веру только потому, что ответ звучит уверенно.

### Шаг 6. Примите или откатите

Если правка точная — принимаете.
Если агент залез лишнее — откатываете лишнюю часть или всю правку.

## Как проверять изменения в Desktop app

По документации Claude Desktop умеет показывать visual diff.

### Как это выглядит для новичка

1. Выполните небольшую правку через Claude Code.
2. После этого откройте область просмотра изменений.
Обычно визуальный diff появляется в интерфейсе сессии как просмотр измененных строк.
3. Посмотрите, какие файлы были изменены.
4. Откройте именно тот файл, с которым вы работали.
5. Проверьте:
- изменился ли только нужный участок;
- не уехал ли смысл;
- не появились ли лишние новые куски текста;
- не был ли изменен другой файл без просьбы.

Если вы не видите diff сразу:
- попробуйте прокрутить окно сессии выше или ниже;
- проверьте, нет ли отдельной кнопки просмотра изменений;
- если работаете не в Desktop, а в редакторе, смотрите inline diff там.

## Как проверять изменения в VS Code

Расширение Claude Code по документации поддерживает inline diffs.

### Пошагово

1. Откройте файл, который был изменен.
2. Найдите участок, где Claude внес правки.
3. Сравните старую и новую версию по смыслу, а не только по красоте текста.
4. Проверьте, что изменение действительно локальное.

Если VS Code показывает diff view:
- откройте его;
- пройдите изменения сверху вниз;
- не спешите сразу соглашаться со всем.

## Очень полезные фразы для контроля

### До правки

```text
Работай только с одним файлом.
```

```text
Измени только этот раздел.
```

```text
Не трогай остальную структуру.
```

```text
Если считаешь, что нужен больший объем изменений, сначала объясни почему, но не расширяй задачу самовольно.
```

### После правки

```text
Покажи, что именно ты изменил и почему.
```

```text
Сохрани минимальный масштаб изменений.
```

```text
Откати все, кроме правок в этом разделе.
```

## Когда правку нужно остановить

Останавливайтесь, если:
- агент начал менять другие файлы;
- локальная правка превратилась в перестройку структуры;
- смысл уехал, хотя вы просили менять только подачу;
- вы уже не понимаете, где заканчивается полезная работа и начинается лишняя.

Остановка — это не провал.
Это нормальный акт контроля.

## Что дают permission modes именно на этапе первых правок

### `Ask permissions`

Лучший режим для первых изменений.
Он замедляет процесс, но резко снижает риск.

### `Auto accept edits`

Полезен позже, когда вы уже уверены, что агент делает только локальные и понятные текстовые правки.

### `Plan mode`

Полезен до правки, когда вы хотите сначала согласовать направление.

## Практическое упражнение

### Цель упражнения

Сделать одну маленькую правку и осознанно проверить результат.

### Что сделать

1. Выберите один короткий раздел в `01-claude-code-first-contact.md`.
2. Если не пользуетесь Git уверенно, создайте backup-файл.
3. Откройте Claude Code и вставьте такой запрос:

```text
Задача:
Упрости язык одного раздела.

Материал:
Работай только с разделом "Что вы сможете после этой главы" в файле 2_lessons/claude-code-book/01-claude-code-first-contact.md.

Ограничения:
Не меняй другие разделы. Не меняй смысл. Не трогай другие файлы.

Готово, когда:
Изменится только этот раздел, а формулировки станут проще и короче.
```

4. Выполните правку.
5. Посмотрите visual diff или сравните файл с backup-копией.
6. Решите:
- оставить правку;
- оставить часть правки;
- откатить правку.

## Как эта глава связана с более глубокой логикой курса

До этого момента вы уже:
- познакомились с агентом;
- научились ставить задачи;
- создали память проекта;
- задали безопасные границы.

Теперь появился еще один слой зрелости:
не просто попросить изменение,
а пройти полный цикл проверки.

Именно из таких циклов потом вырастают:
- repeatable workflows;
- более высокая автономность;
- надежная рабочая система.

## Словарь терминов

**Правка**
Конкретное изменение файла или его части.

**Точка возврата**
Способ вернуться к состоянию до изменений.

**Backup-файл**
Резервная копия файла на случай неудачной правки.

**Diff**
Разница между старой и новой версией файла.

**Visual diff**
Визуальный просмотр изменений в интерфейсе Claude Code.

**Inline diff**
Показ изменений прямо внутри редактора рядом со строками файла.
