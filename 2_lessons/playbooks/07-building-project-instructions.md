# Playbook 7. Как собрать проектные инструкции, чтобы агент не забывал правила работы

**Для кого этот playbook**

Этот сценарий нужен человеку, который:
- уже заметил, что агент в одной сессии работает хорошо, а в другой снова теряет важные правила;
- устал повторять одни и те же вводные вручную;
- хочет сделать проект более устойчивым;
- хочет, чтобы стиль, границы и рабочий порядок хранились не только в голове.

## Когда использовать этот сценарий

Используйте его:
- когда вы уже несколько раз объясняли агенту одну и ту же логику проекта;
- когда в проекте важны стиль, аудитория, ограничения и порядок обновления файлов;
- когда вы хотите уменьшить дрейф между сессиями;
- когда начинаете видеть, что без постоянных инструкций качество плавает.

## Какой результат вы должны получить

В конце сценария у вас должно быть:
- один рабочий файл постоянных инструкций проекта;
- понятная структура этого файла;
- правила, которые агент будет подхватывать в следующих сессиях;
- меньше повторных объяснений на старте новой задачи.

## Главная мысль

Проектные инструкции нужны не для красоты.
Они нужны, чтобы агент не стартовал каждый раз из полупустого контекста.

Если вы постоянно повторяете:
- для кого проект;
- каким языком писать;
- что нельзя менять;
- какие файлы надо обновлять после основной работы;

значит, эти правила уже пора выносить в отдельный постоянный слой.

В терминах книг этот слой обычно ближе всего к проектной надстройке над средой:
не ко всей системе вообще,
а именно к правилам этого конкретного проекта.

## Где такие инструкции стоят в общей системе

Полезно видеть три уровня:

1. базовая рабочая среда;
2. проектные инструкции;
3. короткая стартовая проверка перед задачей.

Если сказать совсем просто:
- базовая среда отвечает за общие настройки и привычки;
- проектные инструкции объясняют, как принято работать именно здесь;
- стартовая проверка перед задачей помогает убедиться, что вы открыли нужную папку и нужные правила вообще подхватились.

Именно поэтому один файл инструкций полезен не сам по себе,
а как часть более устойчивого порядка работы.

## Что именно должно жить в проектных инструкциях

В проектные инструкции стоит выносить не все подряд, а только устойчивые правила.

Хорошо подходят такие блоки:
- цель проекта;
- кто аудитория;
- стиль работы;
- что считать нормальным результатом;
- какие файлы обновлять после крупных изменений;
- когда агент должен остановиться и позвать человека.

Плохо подходят такие вещи:
- случайные идеи на один день;
- временные эмоции;
- длинные рассуждения без практического правила;
- микрозадачи, которые и так живут в `BACKLOG.md`.

## Подготовка

Перед началом соберите:
1. три-пять правил, которые вы уже повторяли агенту больше одного раза;
2. один пример хорошего результата;
3. один пример ошибки, которую надо перестать повторять;
4. список индексных или служебных файлов, которые надо обновлять вместе с основным артефактом.

Если вы делаете это в существующем проекте, выполните буквально такие действия:
1. откройте корень проекта;
2. посмотрите, есть ли уже `AGENTS.md`, `CLAUDE.md` или похожий файл;
3. если файл есть, сначала прочитайте его целиком;
4. выпишите, чего в нем не хватает;
5. только потом просите агента дописывать или пересобирать его.

## Шаг 1. Зафиксируйте, что именно вы повторяете чаще всего

Очень полезный стартовый вопрос к себе:

```text
Какие инструкции я уже устал повторять вручную?
```

Обычно это и есть ядро будущего файла.

Например:
- "аудитория нетехническая";
- "объясняй шаги в интерфейсе буквально";
- "не делай конспект вместо полноценного материала";
- "после новой главы обновляй README и индекс".

## Шаг 2. Соберите инструкции по простому шаблону

Для большинства проектов достаточно такой формы:

```text
Scope
Goal
Audience
Core rules
Workflow rules
Stop rule
Update rule
```

Это не единственно возможный формат.
Но он хорошо работает, потому что:
- быстро читается;
- легко обновляется;
- не превращается в хаотичную простыню.

Если ваш проект уже использует английские названия блоков, это нормально.
Если нет, их можно назвать по-русски.
Главное не язык заголовка, а ясность самого правила.

Если у вас несколько уровней инструкций,
не надо писать один и тот же текст везде.
Общий файл держит общие правила.
Локальный файл в подпапке держит местные особенности.

## Ветка A. Если вы работаете в Codex

### Какой файл обычно нужен

В экосистеме Codex чаще всего роль постоянных проектных инструкций играет `AGENTS.md`.

### Как буквально начать

1. Откройте папку проекта в Codex.
2. Проверьте, есть ли в корне или в нужной подпапке `AGENTS.md`.
3. Если файл уже есть, подайте его агенту как текущую базу.
4. Если файла нет, попросите создать его с простой рабочей структурой.

### Подходящий запрос

```text
Нужно собрать постоянные проектные инструкции.
Файл должен помогать агенту стабильно работать в следующих сессиях.
Собери или обнови `AGENTS.md` с такими блоками:
- scope;
- goal;
- audience;
- core writing rules;
- workflow rules;
- stop rule;
- update rule.
Пиши коротко, практично и без пустой теории.
Не превращай файл в backlog.
Не превращай файл в `BACKLOG.md` с временными задачами.
```

### Что проверить после результата

Проверьте, что в `AGENTS.md`:
- есть реальные рабочие правила, а не общие красивые фразы;
- ясно сказано, что делать после основной работы;
- не перемешаны постоянные правила и временные задачи недели.

## Ветка B. Если вы работаете в Claude Code

### Какой файл обычно нужен

В экосистеме Claude Code таким центральным файлом чаще всего становится `CLAUDE.md`.

### Как буквально начать

1. Откройте проект в Claude Code.
2. Найдите, есть ли в корне `CLAUDE.md`.
3. Если файл уже есть, сначала попросите агента найти в нем пробелы.
4. Если файла нет, попросите собрать минимальную первую версию.

### Подходящий запрос

```text
Нужен постоянный файл проектных инструкций для Claude Code.
Собери или обнови `CLAUDE.md`, чтобы следующие сессии стартовали устойчивее.
Внутри нужны блоки:
- scope;
- goal;
- audience;
- core rules;
- workflow rules;
- stop rule;
- update rule.
Не делай длинный манифест.
Нужен рабочий файл с ясными правилами.
```

### Что проверить после результата

Проверьте, что `CLAUDE.md`:
- помогает реально стартовать новую сессию;
- не дублирует без необходимости весь `BACKLOG.md`;
- держит правила в ясной и короткой форме.

## Шаг 3. Добавьте правило обновления связанных файлов

Это один из самых полезных блоков.

Если у вас проект из нескольких файлов, очень важно явно записать:
- какие файлы надо обновлять после новой главы;
- какие индексы надо синхронизировать;
- какие `SESSION-HANDOFF.md` и похожие файлы надо освежать при длинной работе.

Иначе основной файл будет хорошим,
а навигация и служебный слой проекта начнут отставать.

## Шаг 3.5. Добавьте короткую стартовую проверку

После файла постоянных инструкций очень полезно иметь еще и короткую стартовую проверку перед задачей.

Например:
- та ли папка открыта;
- тот ли раздел сейчас активен;
- тот ли файл ближайший по `BACKLOG.md`;
- нужно ли после правки обновить `README.md` и handoff-файлы.

Эту проверку не обязательно держать в отдельном файле.
Но ее полезно явно сформулировать.

## Шаг 4. Добавьте правило остановки

Хороший файл инструкций должен не только разрешать действия,
но и объяснять, когда надо остановиться.

Например:
- при структурной развилке;
- при конфликте инструкций;
- при риске повредить важную существующую структуру.

Это критично.
Без правила остановки агент легко станет слишком смелым в тех местах, где нужна пауза.

## Шаг 5. Не путайте инструкции проекта и `BACKLOG.md`

Это одна из самых частых ошибок.

Проектные инструкции отвечают на вопрос:

```text
Как здесь обычно работают?
```

`BACKLOG.md` отвечает на другой вопрос:

```text
Что именно сейчас надо сделать?
```

Если смешать их в одном файле, получится документ,
который быстро устаревает и хуже помогает в новых сессиях.

## Шаг 6. Попросите агента вычистить слабые места файла

После первого черновика не спешите считать файл готовым.
Дайте еще один запрос:

```text
Проверь этот файл проектных инструкций.
Найди:
- дубли;
- слишком общие места;
- правила, которые нельзя применить на практике;
- места, где постоянные инструкции смешались с временными задачами.
```

Это обычно улучшает файл сильнее, чем еще один виток украшений.

## Простой шаблон, от которого можно оттолкнуться

```text
## Scope
Какая часть проекта покрывается этими инструкциями.

## Goal
Какой результат проект пытается давать.

## Audience
Для кого делается материал или продукт.

## Core Rules
Каким должен быть стиль и что нельзя нарушать.

## Workflow Rules
Какой обычный порядок работы считается нормальным.

## Stop Rule
Когда агент должен остановиться и не действовать дальше без развилки.

## Update Rule
Какие связанные файлы надо обновлять после крупных изменений.
```

## Хорошие и плохие формулировки

### Плохо

```text
Напиши хорошие инструкции для проекта.
```

### Лучше

```text
Нужно собрать постоянные проектные инструкции.
Смысл файла: чтобы агент не забывал аудиторию, стиль, границы и обязательные обновления связанных файлов.
Сделай рабочий файл по блокам:
- scope;
- goal;
- audience;
- core rules;
- workflow rules;
- stop rule;
- update rule.
Пиши коротко и применимо.
```

## Частые ошибки

### Ошибка 1. Превращать файл инструкций в длинный манифест

Такой файл тяжело применять.

### Ошибка 2. Смешивать постоянные правила с текущими задачами недели

Это делает файл нестабильным.

### Ошибка 3. Не писать, какие файлы надо обновлять после основной работы

Из-за этого разваливается связность проекта.

### Ошибка 4. Не указывать stop rule

Тогда агенту неясно, где его свобода должна закончиться.

### Ошибка 5. Пытаться записать в инструкции вообще все

Сильный файл не максимальный по объему.
Сильный файл содержит только устойчивые правила.

## Практическое упражнение

### Что сделать

1. Возьмите один свой проект.
2. Выпишите 5 правил, которые вы уже повторяли агенту несколько раз.
3. Разделите их на постоянные правила и временные задачи.
4. Попросите агента собрать из постоянных правил `AGENTS.md` или `CLAUDE.md`.
5. После первого черновика отдельно попросите найти дубли и слишком общие места.
6. Оставьте в файле только то, что реально будет полезно в следующей сессии.

## Словарь терминов

**Проектные инструкции**
Постоянный файл с рабочими правилами проекта.

**Scope**
Область проекта, к которой относятся инструкции.

**Workflow rule**
Правило про обычный порядок работы.

**Stop rule**
Правило, которое объясняет, когда агент должен остановиться и не продолжать автоматически.

**Update rule**
Правило про обязательное обновление связанных файлов после основной работы.
