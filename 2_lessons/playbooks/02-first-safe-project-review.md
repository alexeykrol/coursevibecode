# Playbook 2. Как безопасно попросить агента объяснить новый проект без изменений файлов

**Для кого этот playbook**

Этот сценарий нужен человеку, который:
- открыл новый проект и пока не понимает, как он устроен;
- хочет сначала разобраться, а не сразу что-то менять;
- боится испортить файлы первой же сессией;
- хочет получить понятную карту проекта простым языком.

## Когда использовать этот сценарий

Используйте его:
- в первый день знакомства с новым проектом;
- перед тем как давать агенту первую правку;
- когда вы вернулись к старому проекту после долгого перерыва.

## Какой результат вы должны получить

В конце сценария у вас должно быть:
- объяснение, что это за проект;
- список главных файлов или папок;
- понимание, где новичок может запутаться;
- запрет на любые изменения файлов в этой первой сессии.

## Главная мысль

Первая взрослая работа с агентом почти всегда должна начинаться с обзора, а не с вмешательства.

Ваша задача сейчас не "улучшить проект", а получить карту местности.

## Подготовка

Перед началом сделайте три вещи:
1. выберите правильную папку проекта;
2. решите заранее, что эта сессия будет без изменений;
3. подготовьте один короткий запрос на обзор.

Подходящий запрос выглядит так:

```text
Сначала только обзор.
Ничего не меняй и не предлагай правки.
Объясни простыми словами:
1. что это за проект;
2. какие здесь главные файлы и папки;
3. где новичок запутается;
4. с какого файла лучше начать чтение.
```

## Ветка A. Если вы работаете в Codex

### Шаг 1. Откройте проект

Откройте Codex в нужной папке проекта.
Если у вас открыто приложение или IDE, убедитесь, что агент смотрит именно на нужную рабочую директорию.

Что нужно увидеть:
- название проекта или путь к папке;
- чат или рабочую панель Codex;
- отсутствие уже запущенной большой задачи, которая может вас сбить.

### Шаг 2. Сразу задайте границу

В первое сообщение вставьте явную фразу:

```text
Ничего не меняй.
Сейчас нужен только обзор.
```

Это важно, потому что агенту нужно не угадывать ваши намерения, а работать внутри рамки.

### Шаг 3. Попросите простую карту проекта

Полный пример:

```text
Ничего не меняй.
Сейчас нужен только обзор.
Объясни простыми словами:
1. что это за проект;
2. какие здесь главные файлы и папки;
3. где новичок может запутаться;
4. с какого файла лучше начать чтение.
Если чего-то не хватает для уверенного вывода, скажи об этом прямо.
```

### Шаг 4. Зафиксируйте ответ в виде короткой карты

После первого ответа попросите:

```text
Теперь собери это в краткую карту:
- цель проекта;
- 5-7 главных файлов;
- с чего читать;
- какие места самые чувствительные.
Ничего не меняй.
```

## Ветка B. Если вы работаете в Claude Code

### Шаг 1. Откройте проект в Desktop app, VS Code или терминале

Проверьте, что открыт именно нужный проект.
Если у вас Desktop app, посмотрите, что в рабочей области видна нужная папка.
Если VS Code, убедитесь, что в левой панели открыт правильный каталог.

### Шаг 2. Поставьте безопасный режим

Если есть выбор режима разрешений, используйте осторожный режим.
Для новичка на первой сессии подходит `Ask permissions` или просто режим без действий по изменению файлов.

### Шаг 3. Дайте тот же обзорный запрос

Подходящий вариант:

```text
Сейчас только обзор проекта.
Не меняй файлы и не запускай лишние действия.
Объясни простыми словами:
1. что это за проект;
2. какие здесь главные файлы и папки;
3. где новичок запутается;
4. с какого места лучше начать чтение.
```

### Шаг 4. Попросите перевести ответ в маршрут чтения

После первого ответа напишите:

```text
Собери это в маршрут для новичка:
1. что читать первым;
2. что читать вторым;
3. какие файлы пока не трогать.
```

## Что считать хорошим результатом

Хороший результат — это не длинная лекция, а практичная карта.

В хорошем ответе есть:
- человеческое объяснение цели проекта;
- конкретные файлы или папки;
- разумный порядок чтения;
- честное указание на непонятные места.

## Что считать плохим результатом

Плохой результат выглядит так:
- слишком общий текст без привязки к файлам;
- смелые выводы без опоры на проект;
- внезапные предложения сразу все переписать;
- отсутствие ясного маршрута чтения.

## Хорошие и плохие формулировки

### Плохо

```text
Разберись тут.
```

### Лучше

```text
Сейчас нужен только обзор.
Ничего не меняй.
Объясни простыми словами, что это за проект, какие здесь главные файлы и где новичок запутается.
```

## Частые ошибки

### Ошибка 1. Сразу просить правки

Вы еще не понимаете структуру, а уже хотите изменения.
Это рано.

### Ошибка 2. Не говорить явно "ничего не меняй"

Для первой сессии это нужно проговаривать прямо.

### Ошибка 3. Просить объяснение без опоры на файлы

Агент должен смотреть на реальный проект, а не фантазировать по названию.

### Ошибка 4. Не фиксировать маршрут чтения

Обзор без маршрута быстро распадается в голове.

## Практическое упражнение

### Что сделать

1. Откройте один реальный проект.
2. Запустите обзорную сессию без изменений.
3. Попросите карту проекта.
4. Попросите маршрут чтения.
5. Сохраните себе четыре пункта:
- цель проекта;
- главные файлы;
- с чего начинать;
- какие места самые чувствительные.

## Словарь терминов

**Обзор проекта**
Первая безопасная сессия, цель которой понять проект без изменений файлов.

**Маршрут чтения**
Простой порядок, в каком читать файлы проекта, чтобы не утонуть в деталях.

**Рабочая директория**
Папка, внутри которой агент видит проект и работает с ним.

**Чувствительное место проекта**
Файл или узел структуры, где не стоит делать ранние эксперименты без понимания контекста.
